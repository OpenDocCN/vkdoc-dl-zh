# 第一部分 数据编织和数据网格基础

# 1. 数据架构演变

本章介绍了研究数据架构的动机。它概述了数据架构的演变，从传统数据仓库到大数据和数据湖，以及它们的主要特征、价值和挑战。它概述了数据驱动世界中行业的需求，最终导致了数据编织概念的出现。

## 简介

当你回顾过去，数据架构的发展是为了应对现有 IT 解决方案的痛点和新业务需求，通常是在新技术新兴的时期。理解这一演变有助于在组织现有的数据景观中定位新的数据架构趋势，如数据编织（Data Fabric）和数据网格（Data Mesh）。^(1) 为了展示新数据架构的价值，关键是要确定能够从新的数据架构趋势中受益的使用案例，并定义范围明确的项目，以便在合理的时间内为业务带来价值。

自从数据库发明以来，特别是 20 世纪 70 年代具有标准化应用程序编程接口（API）的结构化查询语言（SQL）的关联数据库管理系统（RDBMS）出现以来，就有一种需求，即以结构化的方式组织组织的数据，以便分析数据以回答由业务驱动的疑问。^(2) 到了 20 世纪 90 年代中期，数据仓库的数据架构主要是为了报告和结构化历史数据的分析而开发的，可能结合了不同的数据存储。

在 2010 年代，随着人工智能、物联网（IoT）和边缘计算的广泛应用，对处理大量结构化、半结构化和非结构化数据的需求出现。这导致了具有非常吸引力的承诺的数据湖数据架构，将来自不同来源的数据存储在单一位置，通常以文件形式存储，以适应任何所需的数据消费。作为进一步的发展，数据湖屋（data lakehouse）架构出现，它是数据湖和数据仓库概念的融合，使得在数据湖上可以进行机器学习（ML）和商业分析。^(3)

最近，可轻松获得的处理能力和对数据架构本身元数据的 AI 利用导致了数据编织和数据网格架构和解决方案概念的出现，这些概念通过智能和自动化的系统促进了各种数据管道和云环境的端到端集成。

许多数据架构能力，如数据集成、数据复制、提取-转换-加载（ETL）管道、数据治理等，已经存在很长时间了。在过去，一组特定的这些能力被指定用于所有用例，即使它们并不满足用例的要求。数据编织（Data Fabric）架构承诺根据智能元数据引导使用正确的功能来满足正确的用例。

## 数据架构：价值和挑战

RDBMS 引入了几个功能，这些功能成为了后来数据架构的基础。第一个功能是将逻辑数据模型——关系模型——与数据的物理存储分离。这种分离产生了引入一个元数据目录的需求，该目录描述了数据在表中的逻辑布局及其映射到物理存储，并维护诸如数据大小、数据分布和其他特征等统计数据。

RDBMS 提供功能，通过重新组织数据到定义的顺序来维护数据的物理存储，当数据变化过多导致物理布局和因此数据访问次优时。它还通过利用系统可用能力提供备份和恢复功能来确保数据的可用性。

第二个功能是一个标准化的结构化查询语言（SQL），用于访问数据。如今，SQL 的支持存在于所有编程语言的各个版本中，如汇编语言、COBOL、Java 和 Python。SQL 不断扩展，包括 XML 和 JSON 等数据类型以及新的功能，最近还包括了机器学习（ML）和深度学习（DL）模型的处理。关系数据库管理系统（RDBMS）的两个关键功能成为了企业数据仓库（EDW）架构建立的基础，我们将在下一节中探讨这一点。

## 企业数据仓库（EDW）

有趣的是，RDBMS 最初被用于使用 SQL 和 SQL 生成工具存储数据以供报告使用，作为快速获取结果的一种方式。在 20 世纪 70 年代，与存储在数据集或层次数据库中的数据相比，RDBMS 中的数据访问性能不足以满足事务性应用的需求。随着可用计算能力的提高，大多数事务性系统的数据都迁移到了关系数据库中，这显著提高了应用程序程序员的效率。

以下两个主要业务需求导致了企业数据仓库（EDW）架构的发展：

+   **事务性和报告应用数据库的分离**：如核心银行或航班订票等事务性系统是业务关键性的，与报告应用相比，它们具有不同的可用性和响应时间特性。业务要求将用于事务性和报告应用的数据库分离，以避免影响事务性应用的性能或可用性。

+   **从多个来源整合数据并重构以提高性能和数据可用性**：事务应用中的数据没有存储在最适合报告和分析应用的形式。需要明显地从源（们）复制数据并将其转换成最适合报告和分析的格式。

前述需求导致了一个 EDW^(4)架构，其中数据从事务应用（们）上传到临时区域，然后进行有时复杂的数据清洗和数据转换步骤，在派生数据存储在 EDW 之前。通常，数据在间歇性的临时区域中存储多次。

EDW 架构通过以下特点解决了业务需求：

+   合并和清洗（使一致）来自多个事务源的数据。

+   事务应用和报告应用使用的数据分离。

+   即使源数据没有，也要维护数据历史。

+   将数据转换成适合报告性能的数据模型（使用星型或雪花模式）。转换管道为维度表（例如，时间、销售区域、产品等）提供数据，中间的事实表（们）保持维度之间的关系记录，例如，在某个时间段内某个区域的产品销售。

+   使数据对商业用户可用。

EDW 架构非常成功，几乎每个组织都实施了它。即使不是所有组织，许多组织今天仍在使用 EDW 来满足许多报告需求。

然而，随着业务希望对更多、包括半结构化和非结构化数据有洞察力以做出更好的、数据驱动的业务决策，EDW 架构也带来了新的挑战：

+   ETL 过程也在 EDW 中创造了新的数据质量问题。事务应用持续向它们的数据库（模式变更）添加数据元素。这些新或更改的数据元素需要在 ETL 管道中得到反映。EDW 实施没有充分反映事务应用的数据并不罕见，因为 ETL 过程没有正确更新，或者这些更改完全被遗漏了。

+   EDW 中的数据延迟通常超过 1 天，这对于许多分析应用来说不再是可以接受的。一些报告应用操作的数据可能比一周还要旧。

+   如果数据尚未存在于 EDW 中，或者需要花费数月时间来实施 ETL 流程以添加新的数据元素，那么基于存储在 EDW 中的数据进行分析的新业务请求通常会遭到拒绝。

+   尽管许多事务型应用程序今天根据法律要求必须保留历史数据，或者可以使用 SQL 构造（如双时间戳功能）以较小的开销实现历史数据，但 ETL 流程仍然花费大部分时间来计算历史数据，这增加了数据延迟和数据质量挑战。

+   最后，EDW 非常适合结构化数据。集成半结构化和非结构化数据是一个挑战。

在 Apache Hadoop 开源项目使用多台计算机和大量低成本存储提供的全新技术背景下，这些需求导致了数据湖架构的出现。

## 大数据、数据湖和数据湖屋

商业用户需要访问他们组织的数据库来探索内容、选择和标注，并使用他们的术语访问信息，同时基于数据保护和治理的支撑，例如，一个寻找新活动数据的市场营销人员，这需要临时访问广泛的多种数据源并支持决策制定。

存储和处理大数据需要新的数据架构。大量生成结构化、半结构化（例如，XML、JSON）和非结构化（例如，文本、音频、图片）数据的新数据源无法在 EDW 中处理。

Apache Hadoop 开源项目^(5)的诞生源于处理这些大量数据的需求。它提供了成为许多数据湖实现基础的关键技术：

+   **Hadoop 分布式文件系统** (HDFS)，一个分布式文件系统，为存储在文件系统中的数据提供了非常可扩展和性能良好的访问。

+   **MapReduce**，一种用于大规模数据处理的并行、批处理编程模型。Apache Spark 和 Apache Hive 利用内存技术，根据机器学习算法的需求，提供更快的数据处理速度。

+   **YARN**，一个负责管理集群中的计算资源和 Hadoop 通用库和工具的资源管理平台。

随着公共云解决方案的更广泛接受，今天的数据湖除了在本地 Hadoop 实现之外，还可以使用云存储服务来实现。

数据湖架构通常有一个单一的数据存储库，通常以原始格式存储。它不需要在 EDW 中严格维护的模式定义。相反，用户将自定义模式构建到他们的查询中，这使得数据源集成变得更加灵活。

图 1-3 展示了数据湖架构。左侧是所有可能的数据源，它们使用不同的技术（如流、数据复制和数据集成）被复制到数据湖中。它具有多个数据区，用于不同类型的数据处理。着陆区存储所有传入数据的原始格式。分析区和存档存储用于报告、高级分析和机器学习（ML）的转换数据。数据值可能在数据湖的多个数据区中复制。右侧是数据消费应用，如决策管理系统和预测分析工具。引入了信息治理的概念来管理对数据的访问权限，这些数据在原始数据源中将被严格保护。

许多实施了数据湖项目的组织意识到，“将所有内容存储在一个地方”的模式不仅创造了机会，也带来了新的挑战。*最大的挑战不是创建数据湖，而是利用它所提供的机会*。6

如果不仔细管理，复制大量数据可能导致以下挑战：

+   由于数据湖中的更新延迟，使用过时数据进行分析引入了风险。

+   在越来越多的数据源中理解并解决真实版本（version of truth）的复杂性。

+   随着数据和源数量的增加，存储和集成成本及复杂性增加。

+   将业务暴露于众多监管风险之中。由于原始数据源元数据不断变化，数据治理变得极其困难且成本高昂，从而导致数据湖中的适应变化。

因此，许多组织开始质疑与原始承诺相比，数据湖带来的衍生商业价值。数据湖变成了数据沼泽。EDW 和数据湖的挑战产生了对新的数据管理概念的需求，即数据湖屋，它是一种开放的数据管理架构，结合了数据湖的灵活性、成本效益和可扩展性，以及 EDW 的数据管理和 ACID（原子性、一致性、隔离性、持久性）事务，使得在多种数据结构上实现传统的商业智能（BI）和机器学习（ML）成为可能。因此，数据湖屋结合了数据湖和 EDW 的特点和能力。这通常基于开源组件，如 delta lake，它是一个优化的存储层，为在数据湖屋平台上存储数据和表提供基础。delta lake 是一种开源软件，它通过基于文件的交易日志扩展 Parquet 数据文件，以支持 ACID 事务和可扩展的元数据管理。7

实施数据湖屋作为新的数据和人工智能平台需要新的数据架构，即数据编织。在第十章中，我们进一步阐述了数据编织架构和数据网格解决方案的演变。今天的企业需要一个敏捷的数据架构——数据编织架构——它能够隐藏数据源景观不断增长的复杂性，并使业务用户能够轻松消费数据。以智能和敏捷的数据编织架构为基础，数据网格解决方案方法，其承诺建立以数据产品消费模式的数据市场，是可行的途径。

在下一章中，我们将讨论数据编织架构和数据网格解决方案之间的关系和共存。

# 2. 术语：数据编织和数据网格

本章解释了本书中将使用的核心术语，即*数据编织*和*数据网格*，以及这两个术语之间的关系。我们介绍了术语*数据即产品*或*数据购物*，并提供了对 AI 增强数据编织能力的概述。本章以对数据网格作为一个关键概念的数据产品的描述结束。

## 引言

术语*数据编织*和*数据网格*通常被视为不同的、相互冲突的，或者最多是重叠的数据架构或框架、数据管理概念，或用于发现、探索、治理和消费数据的方法。然而，这些概念相互关联，每个概念都强调特定的必要性或目标。

数据编织（Data Fabric）更侧重于架构基础、技术能力和智能分析，以生成支持更智能、AI 注入的系统（AI-infused system）的主动元数据，从而协调各种数据集成风格，使混合云环境中可信且可重用的数据能够被人类、应用程序或其他下游系统消费。数据编目以生成和利用主动元数据被视为任何数据编织的关键组成部分。

数据网格（Data Mesh）将数据主要视为围绕领域所有者组织，他们创建面向业务的*数据产品*，这些产品可以以*自助*和*数据购物*的方式在分布式消费者、组织以及业务线（LoBs）中进行聚合和消费。将来自不同数据源的数据转换为*数据即产品*是任何数据网格的基本范式。以下章节将进一步描述关键短语*自助*、*数据市场*、*数据即产品*和*数据购物*。

在本章中，我们传达了数据编织的视角，作为构建数据网格解决方案的架构基础，其中具有相应技术能力的数据编织还可以作为其他解决方案的架构基础，例如传统的数据仓库（DWH）解决方案。通过使用术语*数据编织*和*数据网格*，我们进一步指的是结构化和非结构化数据、ETL 阶段以及其他与 AI 相关的工件，例如 ML/DL 模型、管道等。

## 数据编织概念

术语*数据编织*（Data Fabric）由 NetApp 在 2016 年的一份白皮书中提出，作为一个“数据管理愿景……能够无缝连接不同云环境，无论是私有、公共还是混合环境”。NetApp 对数据编织的描述侧重于混合云格局，强调数据安全、治理、集成、应用和服务层等关键方面。然而，它忽略了数据编织的基本功能，例如，主动的元数据驱动方法、AI/ML 注入的任务以及自助交付。

Gartner 将数据编织定义为“一种设计概念，它作为数据及其连接过程的集成层（编织）。数据编织利用对现有、可发现和推断的元数据资产进行的持续分析，以支持在所有环境中设计、部署和利用集成和可重用数据，包括混合和多云平台。”在其数据编织架构中，Gartner 强调关键特性，如嵌入式 AI/ML、语义知识图谱、自动化重复性任务、主动元数据（技术、业务、运营和社会）、动态数据集成、数据编目和自动化数据编排。凭借这些特性，Gartner 的数据编织架构描述相当详尽且处于行业前沿。

Forrester^(6) 将数据编织定义为提供“统一、集成和智能端到端数据平台，以支持新的和新兴用例。它自动化所有数据管理功能 `–` 包括摄取、转换、编排、治理、安全、准备、质量和管理 `–` 使见解和分析能够快速加速用例。”在其文档中，Forrester 强调了所有数据管理功能、治理和合规要求、数据目录、集成、AI/ML、知识图谱以及与主数据管理（MDM）的集成中内置的自动化和智能，这是一套相当全面和完整的数据编织能力。

### 数据编织框架

正如你在第一章中看到的，数据编织是数据架构演化的结果。然而，许多其功能在“数据编织”这个术语被提出之前就已经存在了。例如，数据集成和数据治理几十年来一直是 IT 行业的议程。那么，哪些新的要求和能力脱颖而出呢？

这首先是基于 AI/ML 的所有数据编织领域的增强和自动化，包括基于 AI/ML 的智能集成^(7)，生成要捕获在知识目录中的活动元数据和知识图谱。此外，确保可信的 AI 和将传统数据治理范围扩展到统一 AI 治理是一个相对较新的领域，它远远超出了仅仅关注数据的范畴，包括将 AI 模型和相应的 AI 工件纳入治理领域。最后，数据规则和政策自动生成、发现和执行越来越多地依赖于 AI/ML。

1.  **人工智能生成的数字排放元数据**：所有数据编织任务都会生成额外的数据，例如，日志、数据质量度量、人工智能作品、数据访问路径等。我们称这种数据为数字排放，它可以转化为数字排放元数据，可以进一步分析以获得额外的见解，从而改进相应的数据编织任务。

1.  **基于人工智能的语义丰富**：应用人工智能提供了自动化的丰富功能，以语义知识来上下文化数据，例如，基于知识图谱和其他目录中现有的元数据。语义丰富通过利用这种语义洞察力，为应用程序和商业用户简化并优化数据消费。它还可以进一步简化并优化一些关键的数据编织任务，例如，在特定业务环境中搜索和发现资产。

1.  **人工智能融合实体匹配**：异构和分散的数据景观、系统和应用程序阻止了商业用户匹配核心信息（主数据），例如，人物数据（例如，客户、商业伙伴、员工、公民等）、产品和服务。人工智能融合实体匹配补充了现有的确定性和概率匹配技术，产生了更准确的匹配结果。

## 数据网格概念

与“数据编织概念”部分类似，我们从这个部分开始，简要讨论 Gartner 和 Forrester 的观点。

根据 Gartner 的观点，^(12) 数据网格是一个“针对构建以业务为中心的数据产品的特定目标解决方案架构；它是一个可以在技术无关框架内指导设计的解决方案架构。”在其文章中，Gartner 强调了数据网格的关键要求，如“数据作为产品、分布式数据治理和权限、将数据网格作为解决方案呈现”。此外，Gartner 还具体描述了数据网格和数据编织之间的关系，我们将在下一节中讨论。

根据 Forrester 的观点，^(13) 数据网格“是一种在复杂和大规模环境中共享、访问和管理分析数据的社会技术方法 `–` 在组织内部或跨组织。”在其文章中，Forrester 讨论了数据网格的基本社会技术原则，如“以域为导向、数据作为产品、自助、联邦、计算数据治理。它将数据的软性和业务成果放在首位。”

从本质上讲，数据网格解决方案围绕业务域所有者组织数据，并将相关的数据资产（数据源）转换为分布式业务用户可以从各个业务域或功能消费的数据产品。这些数据产品以自主、去中心化和自助的方式创建、管理和使用。自助功能，我们之前将其称为数据编织能力，使业务组织能够拥有具有数据市场购物特性的数据市场。

图 2-3 是一个数据网格解决方案的概念描述，引用了其关键特征或解决方案设计的关键要求。异构和分散的数据源景观包含构建数据网格解决方案相关的源数据。为了解释图 2-3，我们需要指出实施数据网格的一个特定方面：如图 2-3 所示，它还可以包括核心信息，表示由 MDM 系统管理的数据源 _2（主数据）。这些主数据可能与多个数据产品相关，因此必须由不同的产品所有者发现、访问和消费。因此，一个业务用户（例如，来自 business_domain_B），通常从其 business_domain_B 访问数据产品，也可能需要访问由 data_domain_owner_A 拥有的这些主数据。这种需求在图 2-3 中通过中间数据消费者访问作为数据产品的主数据（由 data_domain_owner_A 拥有）的箭头来表示。

+   **数据源**：用于最终构建数据产品的相关数据可以存储在事务系统中、数据仓库系统中，或者数据湖或数据湖屋中，也可以由应用程序和其他系统管理。它作为创建数据产品的基础。

+   **数据市场**（数据购物）：数据产品需要在数据市场环境中提供给相应的业务用户。这包括搜索、发现和消费*数据作为产品*。

数据网格解决方案的价值在于，它将数据产品的创建任务分配给上游的数据工程师和领域专家，他们最熟悉业务领域和相应的需求。

如前所述，来自其他业务领域的业务用户可能对访问另一个业务领域的数据产品感兴趣。这种跨`–`业务领域的数据产品消费也需要通过数据网格解决方案来实现。

在前面的章节中，数据编织架构和数据网格解决方案之间的关系似乎已经被内在地感知，尽管我们还没有清楚地描述这一点。我们将在下一节中详细阐述这一关系。

## 关系：数据编织与数据网格

在讨论数据网格解决方案时，我们强调了关键解决方案的必要性，例如面向业务的数据产品、分布式数据领域所有权、自助服务能力等。这是在不明确说明实现数据网格解决方案所需的技术能力的情况下完成的。我们的数据网格解决方案设计必要性及关键特征可以被视为实施解决方案的一套指南。然而，是数据编织架构使得数据网格成为可能。换句话说，数据编织架构是实现数据网格解决方案的架构基础。

如您在本章前面所见，数据编织（Data Fabric）由一系列技术能力组成，例如数据概览、自动化元数据丰富、业务术语表、人工智能治理等，以提供数据即产品（data-as-a-product）。使用 Forrester 的术语，*数据网格的社会技术方法*和概念是通过这些数据编织技术能力实现的。根据 Gartner 的观点^(14)，数据编织“可以在不遵循数据网格实践的情况下构建。数据网格必须利用数据编织固有的发现和分析原则。”

在与 IBM 客户合作的过程中，我们经常面临这样的观点，即数据编织和数据网格方法是可以替代的，其中必须做出有意识的“非此即彼”的决定。我们更倾向于同意 Gartner 的观点，即数据编织是构建数据网格解决方案的基础架构。此外，数据编织不仅限于作为数据网格的架构基础；它还可以有目的地调整和设计，以服务于其他解决方案，例如最先进的 DWH 系统。

## 数据产品

数据何时成为数据产品？为了将原始数据转换成可消费的数据产品，需要发生什么？将原始数据转换和构建数据产品始于对原始数据明确的所有权定义，并以数据消费者（例如，商业用户）轻松消费相应的数据产品结束。

构建数据产品是由数据领域所有者**启用**的；然而，构建数据产品本身主要是由数据产品所有者驱动的，这可以是一个营销或客户关怀组织、售后服务团队，甚至是一个个人商业用户。数据产品所有者在整个数据产品构建过程中与数据工程师、数据科学家和其他领域专家合作。

+   **数据领域所有者职责**：数据必须在知识目录中可用并注册，相应的元数据应得到丰富，我们将在第五章节中详细说明。必须通过服务级别协议（SLAs）保证数据的可用性、质量、一致性、完整性和时效性。数据治理——包括数据质量——可以通过强制数据规则和政策来保证。最重要的是，数据领域所有权并不会因为数据消费者访问甚至将数据复制到其他数据存储中而消失；它无论数据位于何处，谁在消费数据产品，出于何种目的，都会持续存在。

+   **数据产品所有者职责**：构建数据产品需要数据易于发现；执行数据概要分析；获得额外的洞察，例如通过语义知识图谱；以及执行必要的数据转换作业。这些任务通过知识目录得到简化，并需要以自助服务的方式进行。数据产品是一个可消费的结果，意味着这些流程的结果，其中原始数据被转换成有意义的业务背景。数据产品所有权伴随着与数据产品的可用性、业务相关性、易于消费、完整性和准确性相关的保证 SLA。从某种意义上说，我们可以将“数据购物”这一术语视为构建数据产品的描述。

基础数据编织架构使得数据操作（DataOps）、模型操作（ModelOps）和人工智能操作（AIOps）能够在整个生命周期内协调数据产品的所需业务和技术调整。

数据产品^(17)基于原始数据，这些数据被转换成有意义的业务上下文，并且易于业务用户消费。它通过知识目录实现，可以由持久的原始数据域和数据产品所有权来表征，这伴随着之前描述的服务水平协议（SLAs）。生产数据产品需要数据编织（Data Fabric）能力和基于 GUI 的应用程序或工具，这些工具具有自助服务功能，简化了生产数据产品和将它们注册到知识目录以供业务消费的步骤。

以下是一个数据产品的关键特征。我们可以将这些特征视为存储在知识目录本身中的数据产品规范；数据产品本身存储在数据市场，并可供消费：

1.  **数据产品负责人**：存在一个明确定义的数据产品负责人，他们指定其业务目标、功能、特征和路线图，以及随时间可能实施的增强功能。

1.  **数据产品描述**：数据产品需要得到良好的描述，类似于任何其他产品，即软件程序或软件产品。这可能包括数据产品的目的或业务意图、版本、法律方面和用途限制、底层原始数据的时效性、使用数据产品的成本等。

1.  **语义相关的数据和人工智能资产**：数据产品由一组语义相关的数据和人工智能资产组成，这可能是一组关系表，也可能是一组管道、ETL 阶段的链、Jupyter 笔记本等。这种语义关系由业务上下文和用途目的定义。识别语义相关的数据和人工智能资产的过程是通过知识目录的元数据丰富功能实现的。

1.  **访问方法**：数据产品必须包括访问方法，例如 REST API、SQL、NoSQL 等，以及数据资产的位置。例如，这可能是一个 URI 端点，由与数据产品关联的预定义 REST API 访问。

1.  **政策和规则**：这包括描述谁可以出于什么目的消费数据产品。它还涉及访问控制、授权、治理等方面，所有这些都由知识目录中定义的数据产品政策和规则管理。

1.  **SLAs**：这些是数据产品负责人和数据产品消费者之间定义的 SLAs，可能包括有关数据产品可用性、性能特征、功能、数据产品使用成本、所需的人工智能模型可解释性、底层数据和人工智能资产的可信度和质量评分等方面的协议。

1.  **定义格式**：数据产品需要使用定义的格式进行描述，例如通过包含或指向相关资产（例如，数据产品描述、数据资产、Jupyter 笔记本、AI 模型）、SLA、访问方法等的 JSON 或 XML 文件（或任何其他数据交换格式）来实现。例如，数据产品可以存储在知识目录中作为 JSON 或 XML 文件。

1.  **目录化**：所有数据产品都需要在知识目录中进行注册，同时需要存储 JSON 或 XML 文档。这些数据产品需要被潜在的数据产品消费者和商业用户进行搜索和发现。

1.  **消费就绪**：数据产品需要准备好供消费，这意味着例如底层的原始数据、AI 资产、数据管道、ETL 转换作业等都是可用和可访问的。

创建数据产品可能还依赖于多个数据领域所有者的原始数据。此外，其他数据产品也可能作为构建另一个数据产品的额外输入。例如，数据科学家可能通过从多个数据领域所有者那里获取原始数据或使用几个可用的数据产品作为输入来开发 AI 模型，其中生成的 AI 模型作为营销组织构建定向营销活动作为数据产品集的输入。因此，我们可以建立数据产品层次结构，其中包含来自其他数据产品、数据和 AI 资产的数据产品依赖关系。

我们需要重新强调，数据产品本身并不存储在知识目录中；它们通过数据产品规范进行注册，这些规范也被视为元数据。因此，数据产品与产品规范的关系类似于其他数据或 AI 资产与其对应元数据的关系。将数据产品规范作为 XML 或 JSON 文件（或任何其他数据交换格式）在知识目录中提供，使得数据产品可搜索、可发现，并且可以被商业用户和其他数据产品开发者使用。

对于数据网格解决方案而言，在组织需求背景下关注数据作为产品，从而遵循分布式和联邦的实施方法，这在某种程度上是有代表性的。然而，这却使得知识目录编排和多个知识目录之间的元数据交换在数据网格的任何实施中都变得至关重要。

例如，这种知识目录编排和元数据交换可以通过 Open Data Platform 倡议（ODPi）Egeria 实现，这是一个基于 Open Metadata and Governance（OMAG）和 Open Metadata Repository Services（OMRS）的开源 Linux Foundation 项目^(19)。ODPi 是一个非营利组织，旨在加速和标准化企业的大数据技术开放生态系统；OMAG 是一个旨在创建一组开放 API、类型和交换协议的项目，以允许所有元数据存储库共享和交换元数据；OMRS 是使元数据存储库能够交换元数据的服务。

这种元数据交换是通过一个 Egeria 开放元数据存储库队列实现的，这是一个使用对等交换共享元数据的服务器集合，如图 2-6 所示。任何符合 Egeria 规范的知识目录都可以加入 Egeria 开放队列，以使用开放元数据和治理格式和接口交换和共享其元数据。

正如我们在本章前面所指出的，数据编织是实现数据网格解决方案的架构基础。因此，在随后的章节中，当我们特别指代架构方面时，我们使用术语*数据编织*；当我们特别强调数据产品（数据即产品）和数据市场方面时，我们使用术语*数据网格*。

# 3. 数据编织和数据网格用例场景

本章介绍了实施数据编织和数据网格的几个用例，这些用例也代表了与业务相关的切入点。几乎每个组织都在进行数据治理和隐私倡议，这使得具有业务需求的人能够跨平台访问企业数据和人工智能工件。其他用例是由混合云数据集成驱动的；对客户、供应商和其他各方进行综合视图的需求，以实现更好的业务成果；以及将可信人工智能集成到业务流程中的开发和集成。

## 引言

许多组织意识到实施数据驱动型业务以保持竞争力的重要性。因此，组织的首席数据官（CDO）面临着创建和实施数据战略的挑战，作为数据驱动型业务的基础。

如第二章所述，数据编织架构和数据网格解决方案的基本组件与典型的数据战略关注领域相一致。因此，围绕这两个概念组件的用例为中心，可以创造一个展示数据编织和数据网格旅程的价值和有意义的切入点的机会。

图 3-1 展示了常见的用例，作为开始实施数据编织/数据网格和随时间进展的旅程有意义的和业务相关的切入点。本书后面的章节提供了对这些四个切入点的更深入处理：

+   **数据治理和隐私**：建立一个自动化和一致的人工智能治理环境，这不仅涉及传统意义上的数据，还包括人工智能成果，例如机器学习模型，这些也需要进行治理。

+   **混合云数据集成**：创建一个跨混合云的数据统一视图，这包括公共和私有云部署，以及传统的本地系统。

+   **客户 360 度视图**：通过将人工智能融入匹配过程，提供客户、供应商和其他各方的全面视图，以确保核心信息的准确性和信心增加。

+   **MLOps 和可信 AI**：实施可信 AI 和 MLOps 以检测和解决已部署和在生产中的 AI 模型的偏差、漂移和恶化质量指标，并且进一步解决 AI 模型的可解释性。

数据战略中的优先级定义了最适用于组织的用例或切入点。在从这些切入点开始您的数据编织和数据网格旅程时，您可以选择一个或甚至几个用例，特别是由于这些场景之间存在某种关系。例如，可信和可解释的 AI 在概念上与 AI 治理相关。

为了确保引入数据编织作为新的数据架构的成功，关键是要很好地界定用例，并定义一个在您的数据和 AI 战略中定义的业务价值的结果。它应该显示数据编织架构与组织现有数据景观（例如，数据湖或传统的 EDW）相比的结果差异。

这听起来可能很显然，但许多数据湖项目被宣布为失败，因为具有良好定义模式的 EDW 结构化数据被移动到数据湖中，并以文件形式存储在 Hadoop 环境中。采用这种方法失去了在关系数据库中优化处理结构化数据的优势以及定义良好的模式信息，即它在星型或雪花模式中的定义。数据消费者必须重新检测之前在 EDW 中易于获取的模式信息。如果相同的分析处理需要更长的时间，并且数据处理能力相同，而数据消费者必须承担数据结构检测的负担，那么很难向业务推销数据湖的好处。

讨论的使用案例通过利用数据编织架构和数据网格解决方案，为业务提供独特的价值交付机会。

即使特定用例的目标各不相同，所有用例的实现都将首先导致构建增强智能数据或知识目录的活动，作为实现这些概念的核心基础，如图 3-2 所示。

以下是与知识目录相关的关键活动：

+   **存储元数据**：在知识目录中搜索、定位和存储元数据和相关工件（业务、技术和管理元数据）。即使最终希望将所有组织数据放入数据目录中，也需要逐步构建，并专注于定义的业务领域或与特定组织相关的选定用例中的数据。医疗行业的例子可以是患者信息。知识目录还应存储与 AI 工件相关的元数据，例如 AI 模型、Jupyter 笔记本、管道等。

+   **定义数据血缘**：定义和查看数据血缘（业务和技术）并执行影响分析。由数据架构实现，如数据仓库（EDW）和数据湖，组织中的数据经常被复制以供不同应用消费。使数据管道可见很重要。首先，如通用数据保护条例（GDPR）2 等法规要求组织在请求时从所有副本中删除个人数据。要执行此法规，需要知道哪些数据管道处理了这些数据以及数据管道执行创建了或更新了哪些副本。其次，它还提供了关于数据质量和数据延迟的指示。从位置 A 复制数据到位置 B 并不是技术挑战。保持来自位置 A 和位置 B 的副本在较长时间内一致才是真正的运营挑战。随着副本数量和数据复制量的增加，问题变得无限复杂。第三，它使数据集成优化和替代方法对数据工程师可见。例如，是否需要并且建议在数据管道中复制原始数据，而不是用派生数据补充或扩展原始数据，这可能是一个更智能的方法？

+   **定义数据和人工智能治理**：定义数据和人工智能治理政策、规则和分类对于打破数据孤岛、允许统一的数据消费和防止数据滥用至关重要。它包括持续监控合规性以及执行数据和人工智能规则和政策，以及确保遵守法规和法律。

+   **增加额外见解，丰富数据**：通过自动化分析，对数据源增加额外的见解。通过添加数据分布、数据质量和业务术语等额外信息来增强元数据，对于提供数据作为服务（DaaS）给数据和数据产品消费者至关重要。例如，对客户记录中*电话号码*属性的数据分析可以显示电话号码在各个数据存储中并不一致维护，因此对消费应用或用户的价值有限。

在静态环境中，通过执行前面的活动来建立知识目录对少量数据源来说是直观的。在现实世界中，数据源的数量和现有副本可能很大，元数据也在不断变化。

设计用于支持数据编织（Data Fabric）和数据网格（Data Mesh）实施的产品功能，提供了以下不同的价值：

+   **任务自动化**：智能地自动化发现、分类和编目，以保持元数据最新。例如，为了支持事务应用中的新功能，向客户记录中添加了新的数据属性。这种元数据更改需要传播到数据目录，并通知消费应用关于更改的通知。

+   **限制数据移动**：将原始数据留在其所在位置，并使用数据虚拟化技术实施替代数据集成技术，以显著减少数据重复。这提高了数据的可用性、延迟和质量。许多复制数据的决定是在特定时间基于产品硬件能力做出的，这些能力今天可能已经过时。在较长时间内保持相对少量的元数据更改的当前性和一致性已经是一个挑战。在更长的时间内保持数以 TB 计的数据副本的一致性在现实中是不可能的。

+   **智能部署 AI 模型**：将 AI 模型部署到智能地交付和编排数据以针对特定用例。例如，营销活动服务通过数据虚拟化更好地访问原始数据源，以确定活动的目标群体，而数据准备服务作为 ML 模型开发的输入可以执行数据转换管道，并将准备好的数据作为副本存储在分析区域。智能信息集成，即 AI 信息集成，支持数据工程师简化他们的任务。

## 自动化和一致的治理

术语*数据治理*用于定义、管理、执行组织数据的访问、隐私、可用性和安全性的流程和责任。它通常包括一系列政策、规则和数据分类以及用于监控和执行合规性的功能。如前所述，我们使用术语*AI 治理*在更广泛的意义上，也包括 AI 工件。

数据治理不是一个新概念。实施集中式数据和 AI 治理的需要存在，以检测不同数据源之间的数据不一致性。例如，一个数据属性，如账户号码，名称相同但具有不同的含义，不能用于组合相关数据。或者，相同的属性在不同的数据源中以不同的名称存储，但没有额外的元数据，数据消费者是无法看到的。

每个数据平台都提供定义和管理数据访问规则和政策的函数。这种孤岛式实施是跨数据源数据消费的痛点。例如，当用户需要访问存储在不同数据平台上的数据时，通常需要为每个平台触发特定的流程来批准和定义所需的数据访问，并且完成时间很长。

监控遵守不断增长的数据隐私和保护法律，如欧盟（EU）的 GDPR，以及在不同数据源之间简单地回答谁在组织中获得了哪些数据访问权限，本身就是问题。因此，许多组织开始建立以以下活动为重点的数据和 AI 治理结构^(4)：

1.  **数据和 AI 识别**：识别数据和 AI 资产，并为组织或整个企业创建一个包含相关元数据的清单——知识目录。

1.  **数据管理政策**：定义数据治理的规则和政策，包括数据的创建、获取、隐私、完整性、安全、分类和质量。

1.  **数据评估**：开发流程来衡量数据和质量、使用情况以及 AI 工件的影响，并监控和优化相关操作。

1.  **数据通信**：建立一个通过知识目录获取数据的源文化，并保持存储在知识目录中的元数据的时效性。

显而易见，通过实施一个智能增强的数据目录作为数据织物架构的基础，可以建立一个数据网格解决方案，从而建立数据和 AI 治理结构^(5)。

任何项目执行如果没有实施和使用正确的产品功能都会非常困难。所选产品应支持组织中的数据源和平台，并提供 AI 增强功能以摄取和自动丰富元数据，使业务用户能够轻松理解、协作、丰富和访问正确的数据，快速建立一个高度自动化和一致的治理环境，并自动在整个组织中保护数据。

数据湖架构可能不是满足这种数据访问需求的良好答案，因为每天将大量数据在不同云服务提供商之间复制到单个数据存储中可能会遇到操作困难（例如，网络延迟和带宽问题，跨许多组件的问题确定）并且成本可能变得难以承受。

在不同的数据源上实施数据布料^(9)，重点关注增强数据集成流程，充分利用 ETL、数据虚拟化和实时捕获，以优化对这些众多不同数据源的访问，这似乎是一种更好的架构，能够满足当今对提高敏捷性和灵活性的需求。

这样的项目将以之前概述的步骤进行，再次在一个定义明确的范围内，例如数据域：

1.  **数据识别**：在混合云数据源中识别与领域相关的数据资产，这对于在混合云环境中成功实施至关重要，因为所有数据源所有者都承担着将数据作为服务提供给数据消费者的责任，因此对将数据源的元数据集成到领域知识目录中有着强烈的兴趣。此外，他们还需要提供一个适用的业务术语表。

1.  **数据管理政策**：数据源所有者需要定义管理该领域数据及其 AI 工件创建、获取、隐私、完整性、安全、分类、质量和消费的规则和流程。

1.  **数据评估**：开发流程来衡量数据和质量、使用情况以及数据与 AI 资产的影响，以及监控和优化相关操作。

1.  **数据即服务**（DaaS）：对于组织来说，提供一个数据消费者市场并通过知识目录获取数据变得至关重要。数据集成管道的自动化以及数据与 AI 资产集成技术（如数据虚拟化）的推荐需要成为这部分应用的一部分。

采用数据布料进行混合云数据集成允许创建跨数据存储的视图，从而在运营应用程序之间实现一致性。这是一个将 IT 基础设施进行整合和简化的绝佳机会，以便在任何地方（本地或任何云）运行并自动化数据操作。关键的区别在于实施智能（智能）信息集成，通过增强数据集成流程充分利用各种数据提供方法，如 ETL、数据虚拟化、复制和流技术，以优化对许多不同数据源的访问。

数据布料提供能力，将质量分析和修复原生地集成到数据管道中，而无需昂贵的下游处理。在混合云环境中构建数据网格解决方案依赖于此类数据布料架构的能力。

通过共同的治理规则、政策和程序支持统一的、端到端的 AIDataOps 生命周期^(10)对于数据消费至关重要。

+   哪些客户也是我的供应商，以便谈判更好的合同条款，以及这些客户接受另一份合同的可能性有多大？

如果所有应用程序都在本地运行，最好是在单一平台上，将客户数据集中到 MDM 解决方案中并集成现有应用程序可能效果良好。然而，将新的数据源集成到现有的 MDM 解决方案中是一项相当复杂的任务，并增加了采用新业务解决方案的时间和成本。

## 解锁可信赖的 AI 概念

AI 持续渗透到业务运营的所有方面，并使新的业务用例成为可能，例如提高业务的盈利能力、增强人类能力和经验。图 3-10 说明了数据织物或数据网格项目的第四个和最后一个用例。

根据 IBM 商业价值研究所（IBV）对 C 级高管的一项调查，^(14) 85%的先进采用者正在通过 AI 降低运营成本，99%的参与者报告使用虚拟代理技术后每接触一次成本有所降低。虽然 AI 有很多前景，但许多人质疑应用 AI 的可信度^(15)。四分之三的高管认为 AI 伦理是竞争优势的来源。确实，许多组织将 AI 伦理纳入其业务指南；他们认为在业务和 IT 流程中应用 AI 伦理实践和行动是一个挑战。

可信和可解释 AI 的要求建议实施 MLOps，连接开发和运营领域，这意味着数据编织和数据网格需要在一个平台上协调 AI 模型的开发，例如，在本地 x86 集群或公共云上，以及在另一个平台上进行评分和推理的部署和运营，例如，在 IBM zSystems 上。

1.  **漂移**：检测和纠正漂移（准确性和数据一致性的下降）

1.  **可解释性**：解释人工智能模型（人工智能模型结果的透明度）

1.  **质量指标**：测量人工智能模型质量指标，包括人工智能模型在操作化时的指标

在数据编织和数据网格之旅中实施可信和可解释的人工智能用例（或切入点）需要在供应商产品中具有不同的功能和特性。更重要的是，例如，如果在人工智能模型生产过程中检测到偏差或漂移，或者某些人工智能模型质量指标正在恶化，则需要自动启动纠正措施——至少尽可能做到这一点——以解决这些问题。这可能包括触发人工智能模型的重新训练、重新验证和重新测试过程，包括收集数据的新版本并执行相应的数据准备任务。

# 4. 数据编织和数据网格业务优势

本章深入探讨了我们在与企业对话中听到的业务驱动因素和痛点。从技术角度，主要是数据工程团队以及消费数据的业务团队的角度，讨论了创建数据布线架构和实施数据网格解决方案的业务优势。我们讨论了与组织内数据管理相关的文化转变，这涉及到数据源所有者、数据工程师和数据消费者的数据整体所有权。

## 引言

在第二章中定义了数据布线，作为数据网格实施的基础数据架构之后，让我们回顾一下支持在组织中实施这种新数据架构的巨大投资的关键业务驱动因素。

总体动机或驱动因素很简单：从组织的数据中获得更多价值，以跟上行业步伐，或者更好地，实现竞争优势。

一个例子是全球性的、颠覆性的大流行病^(1)，它创造了所有领域前所未有的商业挑战。快速识别和解决供应链问题，以及提供所有业务功能在线，无需个人互动，只是导致组织重新审视其当前 IT 环境、定义新的 IT 优先事项并加速相关 IT 现代化项目的几个结果需求之一。

另一个例子是全球公认的气候变化^(2)，它导致自然灾害更加频繁。保险公司需要具备敏捷且复杂的分析企业及公开数据的分析能力，以准确估计政策风险，确保公司能够履行其义务并继续创造利润。具有竞争力的保险政策对保险公司的市场份额有直接影响。

这样的业务需求推动了应用景观的变化，例如，应用程序可能迁移到云服务，或者采用新的应用程序架构，这推动了新的数据架构；它导致组织在数据管理和数据消费方面出现新的要求，这对这两个概念的可持续和抗颠覆实施有深远的影响。

让我们通过一个例子来说明。核心应用程序不断更改数据。在过去，这些事务性应用程序在响应时间和可扩展性方面进行了如此优化，以至于它们没有保留数据更改的历史记录。随着今天处理能力、法规要求和数据库功能的发展，当核心应用程序中的数据发生变化时，仅保留历史记录要高效得多。然而，大多数数据更改都是作为获取-转换-管理数据管道的一部分重新创建的，这使得数据管道更加复杂，通常导致管道处理量超过 50%。

## 数据布料和数据网格对管理数据的团队的好处

第三章描述了实施数据布料架构和数据网格解决方案的基础和入口点。以下是一些技术团队（例如，数据工程师）从实施和责任变更中获得的几个好处，这些好处是对关键驱动因素^(4)的响应）：

+   **更快速、更简单的数据交付流程**：以前，数据工程师需要手动调查适用的数据源；开发数据管道，例如，编写 ETL 作业；并告知数据消费者如何消费派生的（转换和汇总）数据。根据业务优先级，这项工作可能需要几周到几个月的时间。通过创建知识目录并使用注释和分析功能，可以更容易地识别适用的数据源，并使用工具来自动化数据集成以实现更快速的数据交付。打破数据源所有者和数据工程师之间的壁垒，数据工程师从数据消费者眼中数据的“所有者”转变为数据源所有者获取数据的加速推动者。这样，数据源所有者就有动力超越事务性应用的职能和非职能需求，将数据源更好地集成到数据编织和数据网格中，例如向知识目录提供元数据和业务术语表。

+   **数据访问管理努力的减少**：通过知识目录和全球政策执行的自动化，引入全球数据治理规则、政策和流程，显著减少了向数据消费者提供数据访问的努力和时间。它们还允许业务线（LoB）成为数据产品设计、实施和所有权的一部分。

+   **降低维护数据质量标准的工作量**：实施数据编织和数据网格的第一个里程碑是对现有数据源和现有数据管道进行编目。知识编目的一个关键能力是对数据源的质量评估。数据质量通过数据完整性、数据正确性和数据的时效性来表示。例如，电话号码是客户记录的一部分。如果电话号码只在 60%的记录中维护，那么通过区号进行的分析将无法正确反映真实的客户分布。另一个例子是，数据库中的数据需要扩展，因此被移动到另一个列中。ETL 过程没有正确更新，仍然从可能未被删除的旧列中读取数据。这样的问题可能会使派生数据变得不可用。数据消费者可能没有意识到这样的数据质量问题。5 在对原始数据源的数据质量进行评估后，包括数据血缘，使得数据流可由工具消费。否则，隐藏在数千个转换作业中的内容变得对数据工程师可见，从而创造了减少副本数量的机会。毫无疑问，今天的技术可以从源到目标复制大量数据。在实践中，在较长时间内保持数据副本的一致性是一个巨大的运营挑战。因此，减少数据副本总是提高质量，从而提高对数据的信任度。

+   **降低基础设施和存储成本**：数据副本的减少和数据管理工具的整合，以及数据工程师主要责任的变化，直接导致基础设施和存储成本的降低，尽管这对于首席数据官（CDO）和首席信息官（CIO）及其相应组织来说仍然是一个基本目标。

## 数据编织和数据网格对消费数据的业务团队的好处

如前所述，数据编织和数据网格的技术优势可以直接映射到业务团队的好处.^(6) 这些业务成果与实施这两个概念相关：

+   **自助数据购物**：自助服务已在许多领域得到实施。酒店预订是通过门户网站而不是打电话给酒店来完成的。超市的产品结账是通过自助支付站完成的。因此，业务团队成员期望在公司市场门户中购买组织的数据库和 AI 资产的想法并不遥远。具有这些功能的 Data Fabric 提供了实施此类数据和 AI 资产市场的基础设施，可以为业务用户提供无烦恼的全自助数据购物体验。自动执行数据和 AI 治理规则、政策和流程确保业务所有者只能访问他们有权查看和处理的（清晰或掩码）数据。

+   **合规性和安全性**：自动执行数据治理规则和流程不仅是数据市场的前提。首先，数据治理规则、政策和流程需要在组织内一致性地定义，以便自动执行。与今天相比，合规性定义和工具因业务领域、应用程序或甚至负责人员而异，每个平台和系统都有其专有的方式来定义和确保数据安全。因此，尽管使数据和 AI 可用于全面业务分析利用，但数据合规性和安全性实际上得到了提高。

+   **更快和更准确的洞察**：通过降低数据业务消费者的技能水平并通过自助市场简化数据访问，数据消费者可以获得更快和更与业务相关的洞察。此外，通过应用智能和自动化的数据管理以及数据资产和 AI 资产的文档化血缘，显著提高了数据质量、完整性和及时性，这也导致了更准确和与业务相关的洞察。

+   **数据分析所花费的时间**：数据消费者，尤其是数据科学家，现在大部分时间都在寻找和准备数据，而不是专注于数据分析或构建 AI 模型。这种数据探索和准备任务与实际开发时间（包括训练、验证、测试等）的比例通常是 80/20。这种更快、自助访问相关数据和 AI 资产的方式，使得数据消费者，尤其是数据科学家，能够将时间集中在数据分析上，同时也促进了更快地洞察业务问题。

总结来说，基于数据编织架构的数据市场（Data Mesh）允许数据消费者以自助模式更快地发现和访问高质量数据（实现数据即产品化的数据市场），消除了等待数据工程师^(8)准备数据和为业务消费构建 ETL 阶段或管道的潜在长期延迟。

因此，对业务挑战的相关和准确洞察可以更快地获得，并且可以以更高的敏捷性采取行动。

## 关键要点

我们以表 4-1 中总结的几个关键要点来结束本章。

关键要点

| --- | --- | --- |
| 1 | 从组织数据中获得更多价值的动机 | 总体动机或驱动因素很简单：从组织数据中获得更多价值，以跟上行业步伐或甚至获得竞争优势。 |
| 2 | 数据编织架构和数据网格解决方案的关键驱动因素。 | 数据编织或数据网格的关键驱动因素是更快地定位数据、简化数据访问和消费，以及应用一致的数据和人工智能治理。 |
| 3 | 对数据的需要增长速度快于数据交付。 | 以更敏捷的方式从数据中获得可操作的洞察并持续生产数据产品的需求，超过了使用现有数据架构能力进行交付的能力。 |

| 6 | 更快、更准确的洞察。 | 数据编织和数据网格功能使业务分析师能够更快地获得对业务挑战的准确洞察。 |
| 7 | 项目特定层面的业务成果应细化。 | 从技术和业务团队得出的业务成果应以价值驱动因素和度量（KPIs）来表示。 |
