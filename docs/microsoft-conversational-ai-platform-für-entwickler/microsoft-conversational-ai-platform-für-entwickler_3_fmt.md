# 3. Azure 认知服务简介

对话式应用程序和代理通常凭借其智能能力从其他应用中脱颖而出。这些智能能力通常可以分为以下几类：

- 语言
- 语音
- 视觉
- 决策

在每个类别中，都有许多不同的应用场景，聊天机器人或对话式应用可以从中受益——无论是理解用户以文本形式发送的消息的能力，还是将语音转换为文本以供进一步处理的技能。微软提供了一系列名为 Azure 认知服务的 API，使开发者能够为应用程序注入智能。其总体目标是让开发者能够将任何类型的认知能力引入应用程序，使其更智能、更接近人类。这些认知服务的架构允许用户将两个或多个认知服务组合在一起，这对于聊天机器人应用尤其有用。

因此，本书的第三章将详细探讨第 1 章中提到的许多智能工具，这些工具对于使用 Microsoft Azure 平台构建成功的对话式 AI 应用至关重要。不过，重点将明确放在语言类别上，因为它在开发对话式应用时最为重要。

## 语言类别：从非结构化文本中提取含义

语言类别是对话式应用中最重要的方面之一。在开发诸如聊天机器人这类旨在以对话方式与用户交互的应用时，为应用注入语言驱动的智能能力至关重要。虽然语言理解只是其中的一个方面，但对话式应用中还有处理语言的其他方面，例如文本分析或文本翻译。因此，以下部分将详细描述语言类别中部分认知服务背后的概念。

## 语言理解 (LUIS)

如第 1 章所述，在对话式应用中理解自然语言文本是一个基本组成部分，它决定了该应用是否会被人们使用。因此，微软提供了一种名为语言理解（下文简称 LUIS）的 API，它提供了自然语言处理功能。LUIS 的理念是让人们拥有自定义的自然语言理解模型，而无需自行开发必要的算法：

> 语言理解 (LUIS) 是一种基于云的 API 服务，它将机器学习智能应用于用户的自然语言对话文本，以预测整体含义并提取相关的详细信息。

> LUIS 的客户端应用是任何与用户进行自然语言交流以完成任务的对话式应用。客户端应用的示例包括社交媒体应用、聊天机器人和语音驱动的桌面应用。(Microsoft Docs, 2020)

### LUIS 构建块

LUIS 应用中最重要的组件如下，它们对于理解该服务的工作原理至关重要。

从实践角度来看，这意味着每次通过 API 端点调用 LUIS 应用程序以获取给定用户话语的意图和实体时，您都会以识别出的意图和实体的形式获得结果，如图 3.1 和表 3.1 所示。

**表 3.1** LUIS 组件

| 组件 | 描述 |
| --- | --- |
| **话语** | 话语是用户发送给对话应用程序（例如聊天机器人）的非结构化输入文本。一个例子是句子“西雅图的天气怎么样？”。 |
| **意图** | 意图是用户想要执行的特定任务的定义。它可以被视为用户表达的目标或目的。对于话语“西雅图的天气怎么样？”，该话语的意图将是 `GetWeather`。 |
| **实体** | 实体是话语中的文本片段，可用于从非结构化文本中提取特定数据，例如名称或日期等重要值。上述 `GetWeather` 意图中话语的实体将是地点，例如本例中的 `Seattle`。 |
| **分数** | 分数表示 LUIS 应用对特定话语与特定意图匹配或在话语中识别出实体的确信程度。该值是一个介于 0 和 1 之间的数字，接近 1 的预测值表示相对较高的置信度，而接近 0 的值表示 LUIS 对结果准确或正确的置信度较低。 |

**图 3.1** LUIS 基本说明

从开发角度来看，您返回的结果是一个 JSON 表示形式。因此，对于图 3.1 中提到的那个话语，您基本上会得到以下 JSON 字符串作为结果：

因此，LUIS 并非专为聊天机器人设计，也可以应用于其他需要语言理解的场景。然而，由于本书专注于对话式 AI 平台，后续内容将探讨 LUIS 在对话用例中的应用。

### 创建 LUIS 应用程序

在创建 LUIS 应用程序之前，您需要确保拥有一个有效的 Microsoft Azure 订阅。重要的是，您的帐户必须能够在此 Azure 订阅中创建新资源；否则，您将无法执行以下步骤。

**注意**

如果您还没有有效的 Azure 订阅，可以通过此链接 `https://azure.microsoft.com/en-us/free` 注册一个新的 Azure 订阅，并获得 200 美元的免费额度。

创建新的 LUIS 应用程序主要有两种方式：通过基于 Web 的门户和通过 CLI。以下部分将介绍这两种方法。两种方式的结果相同，不过使用 CLI 时，您有更高的机会自动化这些任务。当您需要一次性创建多个 LUIS 应用程序或同时更新多个 LUIS 应用程序时，这将非常有用。

### 使用 CLI

虽然使用 `LUIS` 门户是学习 `LUIS` 概念和功能的好方法，但还有另一种管理 `LUIS` 应用程序的方法，即 Bot Framework CLI。如第 2 章所述，可从 [`aka.ms/bfcli`](https://aka.ms/bfcli) 下载的 Bot Framework CLI 旨在让 Bot Framework 开发者能够通过命令行管理与聊天机器人或对话应用程序相关的所有服务。因此，本章的下一部分将展示如何利用 Bot Framework CLI 来创建、更新、测试和发布 `LUIS` 应用程序。

使用 Bot Framework CLI 创建新的 `LUIS` 应用程序的基础是描述语言理解模型的 `.lu` 文件或 JSON 文件。在我看来，使用 `.lu` 文件的优点是，与 JSON 文件相比，它首先更易读，也更容易创建。因此，您需要在您选择的文件夹中创建一个新文件，使用合适的名称（在本教程中，`03_convAIBook-LUIS` 用作所有 `LUIS` 文件的名称），文件扩展名为 `.lu`，这代表一个语言理解文件。此 `.lu` 文件应基本上代表您在前面步骤中使用 `LUIS` 门户创建的 `LUIS` 应用程序；因此，它将包含以下部分：

- **LUIS 应用程序信息**
- **意图声明**
- **实体定义**

下面是一个示例 `.lu` 文件，描述了 `BookTable` 和 `GetWeather` 这两个意图及其话语和实体：

# LUIS 应用程序信息

```
> LUIS 应用程序信息
> ! # @app.name = ConvAIBook-LUIS
> ! # @app.desc = 用于演示目的的 LUIS 应用程序
> ! # @app.versionId = 0.1
> ! # @app.culture = en-us
> ! # @app.luis_schema_version = 7.0.0
> ! # @app.tokenizerVersion = 1.0.0
> # 意图定义
#### BookTable
- 请预订 {@locationName=redmond steak house} and grill 的一张桌子。
- 明天能在 {@locationName=famous sushi bar} 为 4 人预订一张桌子吗？
- 明天能在 {@locationName=tom's diner} 预订一张桌子吗？
- 请在 11 月 23 日为 5 人在 {@locationName=jamie's kitchen} 预订一个位置。
- 请为六人在 {@locationName=hard rock café} 预订一张周六下午 3 点的桌子。
#### GetWeather
- 你能告诉我雷德蒙德的天气预报吗？
- 纽约的天气怎么样？
- 我想了解阿姆斯特丹天气的一些细节。
- 西雅图的天气怎么样？
- 柏林的天气怎么样？
### 无
> # 实体定义
@ ml locationName
### 预构建实体定义
@ 预构建 datetimeV2
@ 预构建 geographyV2
@ 预构建 number
```

将上述所有部分复制并保存到新创建的 `03_convAIBook-LUIS.lu` 文件中后，您就可以进行下一步了，即将 `.lu` 文件转换为 JSON 文件。导入命令要求传递一个 JSON 文件作为输入参数，该文件包含要创建的 `LUIS` 应用程序的定义。因此，我们需要在安装了 Bot Framework CLI 的命令行中执行以下命令：

```
bf luis:convert --culture "en-us" --in ".\03_convAIBook-LUIS.lu" --out ".\03_convAIBook-LUIS.json"
```

**注意：** 如果您想创建其他语言的语言模型，请确保在 Bot Framework CLI 命令中为您的 `LUIS` 应用程序选择正确的区域设置。

此命令会在当前文件夹中为您创建一个名为 `03_convAIBook-LUIS.json` 的文件，该文件定义了您之前在 `.lu` 文件中描述的 LUIS 应用程序。现在，您可以使用以下命令将此 JSON 文件作为新的 LUIS 应用程序导入到您的 LUIS 帐户中：

```
bf luis:application:import --endpoint "https://.api.cognitive.microsoft.com" --subscriptionKey "yourLUISSubscriptionKey" --name "ConvAIBook-LUIS-CLI" --in ".\03_convAIBook-LUIS.json"
```

# QnA Maker

语言类别中的另一个产品是名为“QnA Maker”的服务。这是一个旨在帮助您以简单 FAQ 风格构建复杂语言模型的工具。QnA Maker 可以被视为一种知识库即服务，因为它允许您以最小的努力使用您的数据创建知识库。然后，这些知识库可以被聊天机器人等对话应用程序利用，以回答用户的问题。因此，本节将涵盖将知识库集成到对话应用程序中所有重要的方面：

> *通过用户友好的界面或 REST API，利用常见问题页面、支持网站、产品手册、SharePoint 文档或编辑内容，创建、训练和发布一个精密的机器人。* (Microsoft, 2020)

## QnA Maker 构建块

如图 3.25 所示，`QnA Maker` 依赖于多种 Azure 服务。最重要的是，所有服务和资源都在您的 Azure 订阅中进行部署。这意味着，存储在您的 `QnA Maker` 服务中并经过处理的数据，都处于您的控制之下。`QnA Maker` 本身使用一个专门的 `QnA Maker` 服务，该服务负责存储您的订阅密钥。这些密钥是与 `QnA Maker` API 通信所必需的。

![](img/495919_1_De_3_Fig25_HTML.png)

图 3.25 `QnA Maker` 架构

您添加到 `QnA Maker` 知识库中的所有数据，都存储在一个专门的 `Azure Search` 实例中。在此资源内，QnA 对及其相关的元数据会被存储和索引。此外，您还可以在知识库中存储特定单词或短语的同义词，这些同义词同样存储在 `Azure Search` 中。

从用户的角度来看，您可以使用 `QnA Maker` 门户来创建和维护您的知识库。在此门户中执行的所有操作，都通过 `Azure App Service` 进行处理，该服务也是您 `QnA Maker` 部署的一部分。此资源包含 QnA 运行时，负责处理 API 请求，例如创建知识库或向您的 `Azure Search Store` 添加新的 QnA 对。`Azure App Service` 的第二部分是 QnA 排名组件。在查询 `QnA Maker` 实例时，该组件会被大量使用，因为它会对 `Azure Search` 实例检索到的结果进行评分并确定置信度值，然后再将这些结果返回给客户端应用程序。

在典型的 `QnA Maker` 部署中使用的最后一个服务是 `Azure Application Insights`。此云服务存储您的 `QnA Maker` 服务的所有重要遥测数据，这些数据可以被查询和分析。一个常见的应用场景是基于用户输入分析最常见的有答案或无答案的问题。

在一个 `QnA Maker` 知识库中，存在一些对其使用至关重要的特定组件，如图 3.26 所示。知识库中存储着 QnA 对。当您的对话应用程序（例如聊天机器人）查询 `QnA Maker` 知识库时，输入内容会与 KB 中存储的问题进行匹配。如果匹配成功，则会将答案连同匹配的置信度值一起作为响应返回。每个 QnA 对还可以包含一个元数据列表。这些键值元数据对可用于过滤特定查询的结果，并从 `QnA Maker` 服务获得更精确的答案。

## 使用 CLI

'''

添加完第一个 QnA 对后，务必为问题添加一些替代表述。这一点很重要，因为使用您对话应用程序的用户在提问时，可能不会总是使用完全相同的措辞。因此，您需要考虑用户可能会如何提出某些问题，这些问题应导向相同的答案，以覆盖所有可能的措辞变体。例如，问题“这本书的标题是什么？”也可能以以下示例句子的形式提出：

- 这本书叫什么？

- 这本书的标题是什么？

- 这本书的标签是什么？

- 您能告诉我这本册子的名称吗？

- 我想知道这本出版物的标题。

- 您知道这部作品的名称吗？

以上所有示例都指向询问这本书的名称，但每个示例又各有不同。由于 QnA Maker 使用模式匹配算法来识别和评定最佳答案，因此构建一个精良且智能的知识库的最佳方法是，尽可能多地添加您能想到的替代表述。这将确保无论用户如何查询，您的对话应用程序都能找到最合适的答案。因此，下一步就是将前面文本中的所有这些示例作为替代表述添加到您的第一个问题中，如图 3.36 所示。

### 使用 Bot Framework CLI 添加数据

要使用 Bot Framework CLI 向 QnA Maker 知识库添加内容，您首先需要一个 `.qna` 文件。该文件应包含您想要添加到知识库中的所有新 QnA 对。以下是一个非常简单的示例，其中仅包含一个稍后将被添加到知识库的 QnA 对：

Stephan Bisser 写了这本书，因为他热衷于一切与对话式 AI 相关的事物。

## 文本分析

语言类别中另一个常用于对话式应用程序的服务是文本分析 API。该服务主要涵盖以下四个用例：

*   情感分析

*   关键词提取

*   命名实体识别

*   语言检测

在开发聊天机器人或其他对话式应用程序时，这些用例中的许多都大有裨益，因为它们增强了自然语言处理功能。由于这是一个即用型 API，因此无需调整此服务内的组件。因此，下一节将详细讨论所有四个用例，以解释其背后的概念；但实际实现将在后续章节中介绍。

### 情感分析

想象一下，您正在构建一个集成到客户服务流程中的聊天机器人，作为处理用户请求的第一道关口。在许多情况下，您的聊天机器人能够回答有限数量的问题，通常是常见问题，但在其他情况下，您的聊天机器人可能无法充分回答特定的用户问题。这是一个关键点，因为当聊天机器人无法回答问题时，您不希望失去用户，因此您需要考虑某种人工转接。人工转接基本上描述了当聊天机器人无法再回答用户问题时，将对话从聊天机器人转交给能够帮助用户的人员的过程。然而，在此过程中，需要决定何时进行转接，因为每次聊天机器人无法回答特定问题时都进行转接并不合理。这基本上就是情感检测可以帮助您找到向人工转接的正确时机的最佳场景。

文本分析 API 中的情感分析功能会为给定的文本提供 0 到 1 之间的情感评分。此功能旨在为您提供关于用户情绪的知识，使您能够判断用户当前是积极还是消极情绪。一个简单的例子是句子“我对昨天在餐厅得到的优质服务非常满意，一定会再来的。”如果您使用这个例句调用文本分析 API 的情感分析功能，您将得到 0.90463 的情感评分，这意味着该句子相当积极，如图 3.53 所示。

![](img/495919_1_De_3_Fig53_HTML.png)

图 3.53

### 关键词提取

关键词提取是文本分析 API 中的另一项功能，可为您的对话应用程序带来一些优势。继续以之前提到的集成到客服解决方案中的聊天机器人为例，想象一下用户向您的聊天机器人发送相当长的消息，例如：“我昨天对餐厅里周到的服务非常满意，绝对会再来的。不过，我不太喜欢汤几乎是凉的，而主菜又烫得没法吃。但冰淇淋甜点非常完美！”这条消息相当长，因此聊天机器人需要分析的不仅仅是一个句子，而是多个句子。这可能是一项艰巨的任务，尤其是在需要转接给人工客服的情况下，因为可能涉及多个专家。

## 翻译器

如第 1 章所述，翻译是聊天机器人应用场景中的一个基本组件，因为它简化了数据管理流程。通过在您的聊天机器人解决方案中引入翻译功能，您无需为每种语言单独创建知识库、语言模型和其他重要服务，因为您基本上可以将所有“后端”组件保持在一两种语言中，并在进一步处理之前，将任何其他语言的输入翻译成这些语言之一。Azure 认知服务中的翻译器服务提供了以下可在对话应用程序场景中使用的功能：

*   翻译文本。

*   音译文本。

*   检测语言。

*   查询单词翻译。

*   确定句子长度。

### 翻译

最常用的功能自然是“文本翻译”功能。通过此功能，您可以将句子从一种语言翻译成超过 70 种语言。使用翻译功能相当简单。由于它是一个 API，在最简单的情况下，您只需创建一个 `HTTP-POST` 请求，提供您的 Azure 订阅密钥以访问 Translator 资源，选择要将文本翻译成的语言，并添加要翻译的文本，如下面的 `curl` 请求所示：

```
curl -X POST "https://api.cognitive.microsofttranslator.com/translate?api-version=3.0& from=en& to=de" -H "Ocp-Apim-Subscription-Key: yourTranslatorKey" -H "Content-Type: application/json; charset=UTF-8" -d "[{'Text':'Hallo, wie ist Ihr Name?'}]"
```

结果是将指定文本翻译成您所选语言，如下所示：

```
[{
"Übersetzungen":[
{
"text": "Hallo, wie heißt du?",
"bis": "de"
}
]
}]
```

此外，您还可以通过将所有语言添加到 HTTP 请求中，同时将文本翻译成多种语言，如下例所示：

```
curl -X POST "https://api.cognitive.microsofttranslator.com/translate?api-version=3.0& from=en& to=de&to=it&to=ja&to=th" -H "Ocp-Apim-Subscription-Key: yourTranslatorKey" -H "Content-Type: application/json; charset=UTF-8" -d "[{'Text':Hi how are you?'}]"
```

作为结果，您将收到所有选定语言的翻译句子：

```
[{
"Übersetzungen": [
{
"text": "Hallo, wie geht es dir?",
"bis": "de"
},
{
"text": "Ciao come stai?",
"zu": "es"
},
{
"text": "やあ、元気。",
"bis": "ja"
},
{
"text": "ไง เธอเป็นไงบ้าง",
"bis": "th"
}
]
}]
```

翻译功能在您需要构建多语言聊天机器人，但不想为每种支持的语言都构建 LUIS 应用程序或 QnA Maker 知识库的场景中非常有用。在这种情况下，您可以选择主要语言，并专门为这些最常用的语言开发语言模型和知识库，然后将其他语言的用户消息翻译成这些语言之一，从而以最小的投入获得最佳效果。

> **注意**：有关 Translator API 的详细语言支持，请访问 [`docs.microsoft.com/en-us/azure/cognitive-services/translator/language-support`](https://docs.microsoft.com/en-us/azure/cognitive-services/translator/language-support)。

### 语言检测

在构建多语言聊天机器人时，在应用自然语言处理功能（如语言理解或关键短语提取）之前，检测用户语言的能力至关重要。因此，Translator API 提供了检测特定句子语言的功能。这在您拥有多个 LUIS 应用程序和 QnA Maker 知识库（支持多种语言）且需要确定特定用户输入应使用哪种语言的场景中非常有用。在这种情况下，您可以使用“语言检测”功能首先确定用户输入的正确语言，然后查询 LUIS 或 QnA Maker 以相应地处理用户输入。以下 HTTP 请求演示了使用此功能的一个非常简单的示例，输入句子为“In welcher Sprache ist dieser Text geschrieben?”

```
curl -X POST "https://api.cognitive.microsofttranslator.com/detect?api-version=3.0" -H "Ocp-Apim-Subscription-Key: yourTranslatorKey" -H "Content-Type: application/json" -d "[{'Text':'In welcher Sprache ist dieser Text geschrieben?'}]"
```

上述 `HTTP POST` 调用的结果是，英语被检测为前述句子的语言。同时，菲律宾语和爱尔兰语也被识别为备选语言：

```
[{

"Alternativen":
[
{

"isTranslationSupported":true,
"isTransliterationSupported":false
},
{

"Punktzahl: 1.0,

"isTransliterationSupported":false
}

}]
```

# 在聊天机器人中组合认知服务的最佳实践

由于您已经单独了解了对话应用场景中最常用的 `Azure Cognitive Services`，下一节将介绍组合其中一些服务的最佳实践。如前所述，这些 `Cognitive Services` 中的每一个基本上都是作为 API 提供的，可以由应用程序（无论是聊天机器人还是其他解决方案）调用；但由于本书的重点是对话应用程序，因此以下场景将仅涵盖对话应用程序。

## LUIS 和 QnA Maker

在创建您的第一个聊天机器人时，最好从一个相对简单的用例开始。将 `QnA Maker` 集成到您的聊天机器人中就是这样一个场景，因为您可以基于 `QnA Maker` 知识库快速构建一个聊天机器人。然而，`QnA` 当然不是复杂的聊天机器人解决方案中唯一使用的场景。因此，当您需要在聊天机器人中涵盖 `QnA` 和基于任务的应用场景时，`LUIS` 是完美的组件。在这种情况下，正确设计您的自然语言处理架构至关重要。实现这一目标的最佳方法是创建一个所谓的 `Dispatch`-`LUIS` 模型，如图 3.59 所示。

这种架构的概念是创建一个顶层 `LUIS` 应用程序，它作为所有底层 `LUIS` 应用程序以及 `QnA Maker` 知识库的 `Dispatch` 模型。当您的聊天机器人收到新的用户请求时，此 `Dispatch` 应用程序用于确定最合适的路由。`Dispatch` 模型基本上充当所有 `LUIS` 模型和知识库之间的路由器。`Dispatch` 模型基本上为每个 `LUIS` 应用程序和知识库包含一个意图。当使用一个表述查询 `Dispatch` 模型时，`Dispatch` 应用程序能够识别匹配的意图，然后您可以使用该意图将您的请求路由到正确的 `LUIS` 应用程序或 `QnA Maker` 知识库，具体取决于该表述是存储在您的某个 `KB` 中，还是与您的 `LUIS` 应用程序的某个意图相匹配。

**注意**

要创建这样一个 `Dispatch` 模型，您需要使用 `Dispatch CLI` 工具，该工具尚不属于 `Bot Framework CLI`，但可以从 [`aka.ms/bot-dispatch`](http://aka.ms/bot-dispatch) 下载。

上面在图 3.59 中概述的示例，是您在为一个有多个团队参与的大型组织构建聊天机器人时可能会遇到的情况。在这种情况下，创建一个涵盖所有可能用例的单一 `LUIS` 应用程序并不总是最佳方法。因此，您可以将您的语言理解模型拆分为多个部分，这使不同团队能够独立处理不同的 `LUIS` 应用程序，而不会影响语言模型的其他部分。

## QnA Maker + 翻译器

构建多语言聊天机器人的另一个最佳实践是使用本章前面描述的 `Translator` API 以及 `QnA Maker`。这在您为旨在面向跨国场景的聊天机器人构建大型知识库时可能非常有益。在这种情况下，您基本上有三种选择：

1.  为聊天机器人要支持的每种语言创建 `QnA Maker` 知识库。

2.  为主要语言创建 `QnA Maker` 知识库，并对所有其他语言使用 `Translator` API。

3.  为聊天机器人的主要语言创建一个 `QnA Maker` 知识库，并对所有其他语言使用翻译器 API。

第一种选择当然是最耗时的，因为这意味着您需要为您想要支持的每种语言构建一个知识库。现在，如果您正在开发一个旨在支持来自世界各地用户的聊天机器人或对话应用程序，您需要构建大量的知识库，然后聊天机器人会连接到这些知识库。在许多情况下，这种实现的工作量太高了，因此很少选择这种方案。

与第一种相比，第二种方案的实现工作量稍小，因为您不需要为每一种语言都构建一个 `KB`，而只需为优先级高的语言构建。当您希望特别针对某些国家，但又不想屏蔽特定语言时，这种方案是最佳选择。在这种情况下，您可以决定哪些语言不应自动翻译，而应由手动构建的知识库来服务。这些语言的用户请求将由针对该特定语言手动构建的知识库来回答，这在大多数情况下会带来更好的结果。

## LUIS + 文本分析

在许多面向客户的聊天机器人示例中，使用 `LUIS` 来识别用户的意图并启动对话中的特定例程或对话。如图 3.59 所示的示例，用户和聊天机器人之间的对话中可能有许多路径，例如获取天气信息或管理餐厅预订。在大多数场景中，仅使用 `LUIS` 可能就足够了，因为它能识别用户的意图并采取相应行动。但特别是在用户感到沮丧或生气的情况下，识别用户的情绪并据此决定聊天机器人是继续对话还是转接给人工客服，可能是一个不错的选择。

因此，将 `文本分析` API 和 `LUIS` 集成在一起以涵盖这些场景可能是一个好方案。如图 3.60 所示，如果对话沿着“愉快路径”进行，聊天机器人可以继续对话并处理用户的请求。相反，在“关键路径”的示例中，当聊天机器人检测到用户消息中的负面情绪分数时，最好将对话转接给人工客服。
