# 4. 聊天机器人的设计原则

机器人的整体设计是聊天机器人成功的最重要因素。作为开发人员，您的目标是开发用户愿意使用的聊天机器人。因此，您需要考虑如何让用户在使用您的聊天机器人而非其他 Web 应用程序或解决方案时获得优势。为了让用户真正使用您的聊天机器人，机器人必须以最快且最简单的方式解决用户的需求，并且这种方式要优于其他解决方案。一个例子是在餐厅预订餐桌的用例。“传统”方法如下：在这种情况下，您的目标必须是简化这一过程，例如实现一个聊天机器人用例，用户只需执行以下步骤：

1. 在智能手机应用或网站上打开聊天机器人。
2. 输入句子：“我想在西雅图的一家意大利餐厅预订周日晚上 6 点、2 位客人的餐桌。”
3. 从可用餐厅列表中选择并确认预订。

在前面的示例中，您可以看到使用聊天机器人的预订过程比在应用或网站上的常规方法要简单得多，也省时得多。因此，用户更可能使用您的聊天机器人而非应用或网站，因为他们可以节省时间，同时仍能获得预期的结果。要创建一个成功的聊天机器人，有许多事情需要考虑；因此，本章将涵盖开发成功聊天机器人的所有重要方面。

## 个性与品牌塑造

机器人成功的关键首先在于其个性。如果您的机器人拥有独特的个性，用户可能会更愿意与您的机器人进行对话，因为这会使整个聊天体验更加人性化和自然。因此，您应该为聊天机器人打造一个品牌。通常，您的品牌应包含以下主要特征，以使其脱颖而出：

- **名称**：不要将您的机器人命名为“聊天机器人 XYZ”，而是给它起一个合适的名字，以便用户能够与您的机器人建立联系。
- **标语**：为您的机器人提供一个简短的标语或介绍语，让人们清楚您的机器人能做什么。
- **形象**：向人们展示您的机器人的外观，无论是类似机器人的图像还是人类的插图，以便人们看到他们正在与谁交谈。

您的聊天机器人本质上代表了您个人或公司在用户面前的形象，因此您必须确保机器人的个性能够反映您的核心价值观。想象一下，您正在为一家银行机构构建聊天机器人，该银行所有在柜台服务客户的员工都穿着西装打领带。在这种情况下，聊天机器人应该与银行的态度保持一致，即礼貌待客并尊重客户。因此，开发一个以幽默风格说话并针对年轻一代的机器人是不合理的，因为在许多情况下，年轻人并非银行的主要目标群体。更合理的做法是让聊天机器人的个性和语言与使用该机器人的公司整体形象保持一致。相反，如果您正在为一家专注于销售生活方式产品的年轻初创公司开发聊天机器人，那么幽默的个性就非常合适，因为该公司也希望以这种方式向用户或客户展示自己。



### 开发聊天机器人之前

在开发聊天机器人之前，您应专注于塑造或至少勾勒出机器人的个性，使其符合预期的应用场景，尤其是要契合其目标受众。在许多情况下，这并非开发者能独立完成的任务，而是需要了解目标受众的业务人员参与，他们能够清晰地描绘出聊天机器人的个性。当然，这个过程在大多数情况下涉及角色定义和关键用户分析，因此这绝非仅由开发者完成的任务。您必须始终牢记，您的聊天机器人是您公司或服务的另一个虚拟代表。因此，它应与您公司或产品的整体形象紧密契合。

通常，人们更倾向于聊天机器人做出人性化而非机械化的回应。因此，密切关注您的目标受众至关重要。例如，如果目标受众是 Z 世代，聊天机器人自然可以使用该世代在与同龄人聊天时习惯的表情符号和内容来回应。相反，如果您的目标受众是婴儿潮一代，那么您的聊天机器人就应该使用这一代人习惯的语言。研究这一点的一个合适方法是使用所谓的“个性卡片”，这些卡片基本上描述了聊天机器人在不同情境下的个性。

**注意：** 此处的首要目标是确保聊天机器人的个性在整个对话过程中保持一致，并且可以由最终用户进行调整。

## 问候与介绍

聊天机器人的一个重要部分是它如何向用户介绍自己。这自然应与聊天机器人的整体个性或品牌形象保持一致，以提供统一的聊天机器人体验。例如，如果您正在开发一个具有友好个性的聊天机器人，您应尝试在对话一开始就表达出这种友好。反之，如果您的机器人应更正式，那么问候语或开场白就应以此为导向。

此外，您必须记住，留下积极印象的机会只有一次。如果用户喜欢聊天机器人的介绍和问候，他们就更有可能继续与它对话。这样，他们就更有可能使用通过机器人提供的服务，或访问您的网站以获取有关您产品的更多信息。因此，由于第一印象至关重要，您必须确保问候语能给用户留下积极印象。如果做到了这一点，您的聊天机器人被实际使用的可能性就会大大增加。此外，您也更有可能通过聊天机器人来推广您的产品。

为了给您一个在对话的问候部分应避免什么的例子，图 4.1 展示了一个可能较差的问候场景。这里最重要的一点是，机器人没有首先问候用户。这显得有点不友好，因此您应始终尝试实现所谓的“主动问候”。这是一个概念，即用户在有机会说“你好”之前就首先被问候。其次，聊天机器人提出了一个相当开放的问题：“我能为您做什么？”这类问题通常会导致用户提问不精确，因为用户不能立即知道该问什么。因此，应避免这个糟糕问候的例子，因为它没有立即展示聊天机器人最重要的能力。因此，用户可能不知道该问什么，并很快关闭聊天。

您还应该加入主动问候机制。这样，您的用户就能更好地了解聊天机器人。如图 4.2 所示，机器人通过一张问候卡片向用户介绍自己。这张卡片基本上包含机器人的名称、图标，以及最重要的聊天机器人的动机。通过这种方式，机器人可以在用户需要提问之前，就向用户传达其优势。如本例所示，机器人还应给用户快速开始对话的机会，这通常通过按钮实现。这样，用户甚至无需在聊天中输入任何内容，而是可以点击按钮来启动对话中的特定对话流程。

问候是对话生命周期中至关重要的一部分，因为它对用户的反应起着决定性作用。请记住，第一印象永远重要。因此，如果机器人留下了良好的第一印象，用户就更愿意继续与机器人对话。这反过来又会提高您机器人的使用率，并最终可能带来更多满意的客户。因此，同样，您机器人的个性也很重要，它也应该在问候消息中体现出来。如果您的用户通过聊天机器人得到了热情且恰当的问候，他们就会使用这个聊天机器人。



### 导航（菜单）

聊天机器人对话的另一个重要组成部分是对话导航。与应用程序或网站不同，聊天机器人没有汉堡菜单或导航栏，无法让用户跳转到对话中的特定位置。因此，您需要考虑如何引导用户完成对话。此外，还应考虑对误解或死胡同的管理。因此，整合一种由机器人告知用户的导航策略是一个好方法。例如，图 4.3 展示了一个很好的方法，即通过让用户直接回复“帮助”来引导他们如何在对话中获得帮助。这样，用户在对话的第一条消息中就能看到这一点，并知道如果遇到不清楚的地方或不知道如何继续，只需输入“帮助”即可获得支持以继续对话。

## 对话流程

一个对话通常由多个对话组成，这些对话共同构成一个对话树。这样的对话树本质上是对聊天机器人用例的表示。在实践中，在开始实际开发或创建机器人之前，先对对话树进行建模是一个好方法。这样，您就可以以流程图或类似选项的形式在纸上勾勒出机器人的行为和例程，这些将作为后续开发阶段的对话设计稿。如第 2 章所述，对话是聊天机器人的核心，因为它们是用户与机器人对话时直接提供的对话机制。在设计对话流程时，您还应考虑如何正确处理中断。由于用户可能在对话中途想要切换上下文，您也应考虑如何处理此类情况，并在中断处继续对话。

如图 4.6 所示，聊天机器人的对话树可以由许多不同的对话组成。每个对话框负责处理不同的用例或对话部分。在 Microsoft Bot Framework 中，顶层对话框被称为 `RootDialog` 或 `MainDialog`。它本质上是用户与聊天机器人之间任何对话的起点。`RootDialog` 通常包含一个 LUIS 识别器，负责确定用户的意图，以触发特定的子对话框。

## 自适应对话

在 Bot Framework 中，目前有两种实现对话的概念：一种是许多 Bot Framework SDK 版本中使用的对话库，另一种是新增的自适应对话概念。这两种概念的主要区别在于，自适应对话在处理对话事件（如中断或上下文切换）时提供了更大的灵活性。自适应对话背后的主要思想是提供一种更基于事件的方法来创建对话，该方法既包含对话部分（如语言理解和生成），也包含对话内的操作。由于这种新的声明式模型旨在为开发人员提供一系列构建聊天机器人的模式，而不一定需要通过编写代码来实现，因此本书的重点主要放在描述这种自适应对话方法上。

**注意**

如果您想了解更多关于 Bot Framework SDK 中对话库的信息，请访问 [`docs.microsoft.com/en-us/azure/bot-service/bot-builder-concept-dialog`](https://docs.microsoft.com/en-us/azure/bot-service/bot-builder-concept-dialog)。

自适应对话包含一系列称为触发器的事件处理程序，如图 4.10 所示。触发器本身可以包含一个条件，用于确定何时触发该触发器。每个触发器内部包含一个或多个在触发器执行时运行的操作。事件处理程序基本上处理对话中发生的所有事件，并评估应执行哪个触发器。

![](img/495919_1_De_4_Fig10_HTML.png)

### 自适应对话触发器

Bot Framework 包含多种不同的触发器类型，用于特定场景，如表 4.1 所示，涵盖从识别事件、对话和活动事件到消息事件。

表 4.1

自适应对话 – 触发器类型 (Microsoft, 2020)

| 触发器类型 | 描述 |
| --- | --- |
| **基础触发器** | `OnCondition` 触发器是所有触发器派生自的基础触发器。在自适应对话框中定义触发器时，它们被定义为一组 `OnCondition` 对象。 |
| **识别事件触发器** | 识别程序从用户输入中提取有意义的信息（如意图和实体），并发出事件。例如，当识别器从特定用户表述中提取出意图（或实体）时，会触发 `recognizedIntent` 事件。您可以使用 `OnIntent` 触发器来处理此事件。此类型下可用的触发器有：`OnChooseIntent`、`OnIntent`、`OnUnknownIntent`、`OnQnAMatch` |
| **对话事件触发器** | 对话触发器用于处理与“对话生命周期”相关的特定对话事件。目前 Bot Framework SDK 中有六个对话触发器，它们都派生自 `OnDialogEvent` 类。此类型下可用的触发器有：`BeginDialog`、`CancelDialog`、`EndOfActions`、`Error` |
| **活动事件触发器** | 使用活动触发器，您可以为来自客户端的每个传入活动分配操作，例如当新用户加入且机器人开始新对话时。有关活动的更多信息，请参阅 Bot Framework 的活动架构。所有活动事件都具有基础事件 `ActivityReceived`，并根据其活动类型进一步细化。所有活动触发器派生自的基类是 `OnActivity`。此类型下可用的触发器有：`ConversationUpdate`、`EndOfConversation`、`Event`、`Handoff`、`Invoke`、`Typing` |
| **消息事件触发器** | 消息事件触发器允许您响应任何消息事件，例如消息被更新（`MessageUpdate`）或删除（`MessageDeletion`），或者有人对消息做出反应（`MessageReaction`）（例如，常见的消息反应包括点赞、爱心、大笑、惊讶、悲伤和愤怒）。消息事件是活动事件的一种，因此所有消息事件都具有基础事件 `ActivityReceived`，并根据活动类型进一步细化。所有消息触发器派生自的基类是 `OnActivity`。此类型下可用的触发器有：`Message`、`MessageDeletion`、`MessageReaction`、`MessageUpdate` |
| **自定义事件触发器** | 您可以通过向任何触发器添加 `EmitEvent` 操作来发出自己的事件；然后，您可以通过定义自定义事件触发器，在机器人任何对话框的任何触发器中处理此自定义事件。自定义事件触发器是 `OnDialogEvent` 触发器，当您将其 `Event` 属性设置为与 `EmitEvent` 的 `EventName` 属性相同的值时，它将成为自定义触发器。此类型下可用的触发器有：`OnDialogEvent` |

识别事件触发器包含用于确定用户意图和实体的识别器。当机器人内的多个识别器检测到多个意图时，会触发识别事件触发器中的 `OnChooseIntent` 触发器。因此，此触发器控制用于选择正确意图以继续对话的逻辑。此触发器类型中的 `OnIntent` 触发器在检测到特定意图时执行，以便可以执行定义的操作。相比之下，`OnUnknownIntent` 触发器处理任何 `OnIntent` 触发器均未检测到意图的情况。识别事件触发器中的最后一个触发器是 `OnQnAMatch` 触发器。每当 `QnAMakerRecognizer` 接收到 QnAMatch 意图时，就会触发此触发器，这意味着检测到用户输入与 QnA Maker 知识库中的 QnA 对之间存在匹配。

对话事件触发器类型由用于处理对话生命周期中特定事件的触发器组成。`OnBeginDialog` 触发器包含在特定对话框开始时执行的操作。`OnCancelDialog` 触发器用于防止对话框被取消。`OnEndOfActions` 触发器定义了在对话框的所有操作运行完毕后应执行的操作。而 `OnError` 触发器则处理对话框内的特定错误情况。

在活动事件触发器中，当用户开始与机器人对话时，会执行 `ConversationUpdate` 触发器。当用户与机器人之间的对话结束时，会执行 `EndOfConversation` 触发器。`Handoff` 触发器用于执行人工交接操作，以便将人工接入与用户的对话中。`Typing` 触发器用于判断用户是否正在输入。

消息事件触发器类型中的 `MessageReceived` 触发器用于在机器人收到用户的新消息时执行操作。相比之下，当用户的消息被删除时，会执行 `MessageDelete` 触发器。例如，如果用户对某条消息点赞，则会执行 `MessageReaction` 触发器来处理此类情况，并据此执行必要的操作。`MessageUpdate` 触发器可用于处理用户消息的更新。

## 用户体验

聊天机器人的用户体验在某种程度上不同于“传统”的应用程序或网站的用户体验，因为您创建的是对话式用户界面（简称 `CUI`），而非图形用户界面（简称 `GUI`）。由于人们拥有不同类型的对话习惯，设计聊天机器人的用户体验有时可能是一项挑战。因此，正如本章节中提到的，了解聊天机器人的目标受众至关重要。年轻人习惯于使用 `Facebook`、`WhatsApp` 和 `Slack` 等对话平台。他们习惯于通过即时通讯和聊天与他人交流，而年长一代可能不习惯这类通讯平台，因为他们成长于不同的时代，有着不同的沟通方式。年轻人熟悉在对话中使用表情符号以及图片和视频等富媒体，而年长一代可能并非如此。因此，在开发聊天机器人之前，您应该对目标受众有清晰的认识。

不过，总的来说，开发一个不仅仅通过纯文本消息与用户交流的聊天机器人是一个好的实践。当然，这要求您的聊天机器人所运行的渠道也支持其他类型的消息和富媒体附件。例如，用于短信（`SMS`）的机器人只能发送和接收文本消息，而不能发送图片和视频。然而，`Azure Bot Service` 支持的大多数通信渠道都支持富媒体附件。因此，您的聊天机器人应设计为包含这些对话机制，以积极影响对话流程。

良好的聊天机器人用户体验还包括状态管理的概念。状态基本上是机器人的大脑，用于让机器人能够检索与用户的对话历史。这对于设计良好的对话流程至关重要，因为如果机器人知道中断发生的位置，以及如何在发生中断或上下文切换的点恢复对话，它就能更好地处理中断。此外，通过适当的状态管理，您有机会开发出更人性化的对话行为，因为机器人可以记住与用户之前对话中的特定信息，这同样能改善对话流程。在我们的示例中，聊天机器人处理餐厅预订请求，这也包括预订的管理。因此，如果用户无需提供餐厅预订的完整信息即可更新或取消预订，那将非常有益。相反，机器人应该能够记住用户的预订，并仅通过询问预订参考号来继续处理用户的请求。因此，状态管理引入了机器人记住先前对话信息的能力。

## 富媒体附件

除了机器人能够记住先前对话的部分内容之外，良好用户体验的另一个重要因素是除了文本之外，还要包含富媒体附件。如前所述，这取决于您的机器人部署在哪个渠道上，但总的来说，在用户对话中融入图片、链接、视频甚至表情符号等对话元素是一个好方法。因此，`Microsoft Bot Framework` 支持多种不同的选项来向机器人消息添加富媒体附件。在 `C#` 机器人中，您可以使用以下代码片段，通过向消息添加内联附件，轻松地向用户发送包含图片的消息：

```
msg = MessageFactory.Text("Here you see the dialog tree!");
msg.Attachments = new List() { GetImageAttachment() };
private static Attachment GetImageAttachment()
{
var imgPath = Path.Combine(Environment.CurrentDirectory, @"img", "dialogTree.png");
var img = Convert.ToBase64String(File.ReadAllBytes(imgPath));
return new Attachment
{
Name = @"img\dialogTree.png",
ContentType = "image/png",
ContentUrl = $"data:image/png;base64,{img}",
};
}
```

重要的是，富媒体附件能够增强机器人与用户之间的对话。当您想要展示菜单卡等特定内容时，使用包含图片和其他媒体的消息进行回复是有益的。当然，您也可以向用户发送包含菜单信息的文本消息，但在大多数情况下，菜单的图片会更专业，并且尤其对用户更友好。

### 建议操作

与卡片中用户可点击以继续对话的按钮不同，建议操作是显示在聊天输入框附近的按钮，在用户选择了其中某个按钮或输入文本后就会消失。这些建议操作是一种很好的方式，让用户可以通过点击按钮来回答问题，而无需输入文本。建议操作相对于卡片中按钮的主要优势在于，卡片中的按钮会一直保留在聊天窗口中。因此，用户可能会再次点击来自较早消息的按钮，这可能导致中断或上下文切换，因为机器人可能已经处于对话树的不同部分。使用建议操作，作为机器人开发者的您无需考虑这种情况，因为一旦用户继续对话，建议操作就会从聊天窗口中消失。建议操作的示例如图 4.14 所示。

### 自适应卡片

自适应卡片背后的理念是，只需创建一次卡片，然后让机器人所部署的渠道原生渲染该卡片。英雄卡片和其他卡片类型通常是机器人代码库的一部分，而自适应卡片则使用 JSON 创建。因此，卡片本身会以 JSON 卡片格式传递给渠道，这使得渠道能够原生渲染卡片，同时使用与该渠道内其他部分相同的 UI 样式。目前，表 4.5 中列出的渠道支持自适应卡片，这远非所有 Azure 机器人服务渠道。因此，在决定卡片类型之前，您应该首先了解您打算将机器人部署到哪些渠道。

**表 4.5** 自适应卡片渠道支持 (Microsoft, 2018)

由于自适应卡片本质上是 JSON 文件或对象，因此创建卡片很简单。甚至还有一个自适应卡片设计器，您可以直接在浏览器中设计和创建自适应卡片，而无需编写 JSON 表示法。在自适应卡片中，您基本上可以插入文本块、图像、视频、列、按钮以及输入对象，例如文本输入字段、数字或日期输入字段或选择输入。

**注意**

可用于创建自适应卡片的自适应卡片设计器位于 [`adaptivecards.io/designer/`](https://adaptivecards.io/designer/)。
