# 5. 构建聊天机器人

在上一章中，您学习了如何设计聊天机器人，包括对话流程和用户体验。本章将介绍如何使用名为“Bot Framework Composer”的工具来构建聊天机器人，该工具是 Bot Framework 生态系统的一部分，旨在通过图形用户界面创建聊天机器人，而不是通过代码编写聊天机器人。此外，本章还将详细阐述前面章节中涉及的主题在构建聊天机器人的实际场景中的应用，例如使用语言理解和 QnA Maker 来改进自然语言处理流程。此外，本章还将描述如何使用自定义代码将特定功能集成到使用 Composer 创建的聊天机器人中，这些功能默认情况下不可用。



## Bot Framework Composer 简介

`Bot Framework Composer` 是一款采用图形用户界面的工具，专为创建聊天机器人而设计。`Composer` 面向开发者和高级用户，提供了创建、测试和发布聊天机器人所需的众多功能。例如，`Composer` 还提供了创建聊天机器人核心对话流程之外的关键部分的能力，例如语言理解模型、语言生成模型或 `QnA Maker` 知识库。因此，它可以被视为一个工具，无需编写代码即可提供设计和开发聊天机器人所需的主要功能和能力。

`Composer` 的设计初衷不仅是为了方便开发者，也是为了方便非开发者设计和构建机器人。当然，可视化画布功能加速了机器人开发的整个生命周期，特别是对于那些不经常创建聊天机器人的人来说。使用 `Composer` 基本上有两种方式：两种方案各有优缺点，但两种版本的总体外观和功能是相同的。因此，本书中引用的所有示例均基于 `Bot Framework Composer` 的桌面版。`Composer` 包含一个图形用户界面，如图 5.1 所示，为您提供了一个精良的创作界面，您可以在其中创建对话、触发器和操作。

使用 `Composer` 创建机器人时，与 `Bot Framework SDK` 存在一些差异，机器人开发者需要注意。首先，`Composer` 基于第 4 章中描述的自适应对话，这与 `Bot Framework SDK` 中使用的对话库不同。但同时，这也是 `Composer` 的优势之一，因为自适应对话以 `JSON` 格式存储，因此可以在许多不同的机器人或工具中重复使用。`Composer` 相对于 `SDK` 的另一个巨大优势在于，它将最常用的工具整合到了一个用户界面中。过去，机器人开发者需要访问 `LUIS` Web 门户来管理所有语言理解组件。要管理 `QnA` 问答对，还需要访问 `QnA-Maker` 门户。而对于机器人开发本身，则需要使用 `Visual Studio` 或 `Visual Studio Code` 来开发机器人的逻辑。随着 `Composer` 的推出，这些体验被整合到一个工具中，为开发者在设置和维护机器人所需的关键组件方面节省了大量时间。

### 存储

在自适应对话中，存储基本上被视为机器人的大脑，如第 4 章所述。因此，您可以使用存储来保存特定值，这些值可以在对话过程中的稍后时间点访问，例如用户的姓名或地址信息。在第 4 章中，我们了解到存在以下存储范围，它们提供不同的访问方式和持久性：这些范围用于在对话中存储不同的值，以便稍后重用。您在特定操作中或作为机器人提问结果设置的每个属性，都将存储在上述某个范围中。如果您需要存储关于用户或对话的数据，则应使用用户范围或对话范围，因为这些范围在机器人的整个生命周期内都有效。用户范围主要用于存储用户信息，例如姓名、位置或其他用户属性和设置。对话范围主要针对对话属性，例如对话的元数据（例如开始日期等），这些属性可能由进入同一对话的多个用户共享（例如 `Microsoft Teams` 群聊）。如果您需要存储短期值，则对话范围和轮次范围更为合适。对话范围的生命周期基本上与对话绑定，即当对话结束时，在该范围中定义的属性将被丢弃。轮次范围甚至比对话范围更短暂，因为在该范围中定义的属性仅在轮次内保留，而轮次本质上是由机器人处理的单条消息。

-   对话范围
-   对话范围
-   轮次范围
-   设置范围
-   此范围
-   类范围

这些范围用于在对话中存储不同的值，以便稍后重用。您在特定操作中或作为机器人提问结果设置的每个属性，都将存储在上述某个范围中。如果您需要存储关于用户或对话的数据，则应使用用户范围或对话范围，因为这些范围在机器人的整个生命周期内都有效。用户范围主要用于存储用户信息，例如姓名、位置或其他用户属性和设置。对话范围主要针对对话属性，例如对话的元数据（例如开始日期等），这些属性可能由进入同一对话的多个用户共享（例如 `Microsoft Teams` 群聊）。如果您需要存储短期值，则对话范围和轮次范围更为合适。对话范围的生命周期基本上与对话绑定，即当对话结束时，在该范围中定义的属性将被丢弃。轮次范围甚至比对话范围更短暂，因为在该范围中定义的属性仅在轮次内保留，而轮次本质上是由机器人处理的单条消息。

![](img/495919_1_De_5_Fig2_HTML.png)

*图 5.2 在 Composer 中设置属性*

在前面的示例中，属性值是“硬编码”的，并且类型为 `String`。但这并不总是足够的，因为您可能需要基于某个操作生成值。这个概念在 Composer 中被称为“表达式”。表达式通常用于基于计算函数来计算值。这些表达式可以包括：

- 算术运算，如加法 (`+`)、减法 (`-`)、乘法 (`*`) 或除法 (`/`)
- 比较运算，如等于 (`==`)、不等于 (`!=`)、大于 (`>`) 或小于 (`<`)
- 逻辑运算，如与 (`&&`)、或 (`||`) 或非 (`!`)

此外，还有许多预构建的函数可用于处理各种对象类型，如 `Strings`、集合或日期，以及用于比较和类型检查的操作。

### LU（语言理解）

如前几章所述，语言理解几乎是每个 AI 解决方案中的核心功能。每当您需要处理超出简单 QnA 模式的特定用户请求时，您都需要实现某种形式的语言理解，以便相应地处理用户输入。在 Bot Framework SDK 项目中使用 LUIS 与在 Composer 中使用 LUIS 的主要区别在于，您无需在 IDE 和 LUIS 门户之间切换，因为 Composer 内置了一个内联编辑器，用于创建意图和实体等 LUIS 组件。

在 Composer 中维护语言模型的原理与第 3 章所述基本相同。使用 Composer 的内联编辑器，您可以编辑一个 `.lu` 文件，Composer 会使用该文件自动管理 LUIS 模型。这些 `.lu` 文件通常包含意图、实体和表述。所使用的表示法如表 5.1 所示。

**表 5.1 Composer `.lu` 文件表示法**

| 类型 | 表示法 |
| --- | --- |
| **意图** | `# IntentName` |
| **表述** | `- 一个表述示例` |
| **实体** | `{entityName=example}` |

因此，第 3 章中的示例意图 `GetWeather` 的 `.lu` 文件可能如下所示：

```
#### GetWeather
- 您能告诉我雷德蒙德的天气预报吗？
- 纽约的天气怎么样？
- 我想了解阿姆斯特丹天气的一些细节
- 西雅图的天气怎么样？
- 柏林的天气怎么样？
@ 预构建 geographyV2
```

您会注意到，我们向此意图添加了预构建实体 `geographyV2`，这使得 LUIS 能够自动从用户的表述中识别并提取地名。在 Composer 中使用的 `BookTable` 意图如下所示：

```
#### BookTable
- 请帮我预订 {@locationName=红狮牛排馆} 烧烤店的一张桌子。
- 你能帮我预订明天在 {@locationName=著名寿司吧} 的四人桌吗？
- 你能帮我预订明天在 {@locationName=汤姆小餐馆} 的桌子吗？
- 请帮我预订 11 月 23 日在 {@locationName=杰米厨房} 的五人位。
- 请帮我预订周六下午 3 点在 {@locationName=硬石餐厅} 的六人桌。
@ ml locationName
@ 预构建 datetimeV2
@ 预构建 number
```

在 Composer 中使用这两个 `.lu` 文件的详细信息将在本章的后续部分中介绍，同时还会提供一些关于如何相应处理语言理解的更多信息。



### LG（语言生成）

虽然语言理解为机器人开发者提供了一种简单的方式，为聊天机器人添加自然语言处理功能，但语言生成主要用于赋予聊天机器人特定的语气，使其看起来更智能。因此，语言生成旨在为开发者提供灵活性，以便为聊天机器人的消息添加变化。这不仅提供了将业务逻辑与表示层分离的可能性，还允许在聊天机器人解决方案中使用一致的概念来管理和维护语言生成组件。`.lg` 文件格式相当简单，在某种程度上类似于本章前面描述的 `.lu` 文件格式，并在表 5.2 中进行了描述：

表 5.2

Composer 的 `.lg` 文件格式

| 组件 | 表示法 |
| --- | --- |
| **注释** | `> 这是一条注释` |
| **模板名称** | `# TemplateName` |
| **变体** | `- 这是一条机器人回复` |

```
> 包含 3 个变体的问候模板。
#### 问候
- 你好
- 嗨
- 您好
```

在 `.lg` 文件中，您还可以使用条件语句，根据特定属性来定义回复。例如，问候语也可以包含“早上好”或“晚上好”等词语。默认情况下，机器人会随机选择问候模板中存储的短语之一，这可能导致在晚上 8 点时，机器人回复“早上好”。在这种情况下，您可以通过直接在 `.lg` 文件中使用 `if-else` 语句来解决此问题，以判断“早上好”是否合适，或者用“晚上好”来问候用户是否更好，如下所示：

```
> 带条件的问候语
#### ConditionalGreeting
- IF: ${time == 'morning'}
- 早上好
- ELSEIF: ${time == 'afternoon'}
- 下午好
- ELSEIF: ${time == 'evening'}
- 晚上好
- ELSE:
- 你好
- 嗨
- 您好
```

同样，可以在模板内使用 `switch case` 语句来实现，以确定哪个 `case` 子句符合条件，并相应地设置机器人的回复：

```
> 带条件的问候语
#### ConditionalGreeting
- SWITCH: ${time}
- CASE: ${'morning'}
- 早上好
- CASE: ${'afternoon'}
- 下午好
- CASE: ${'evening'}
- 晚上好
- DEFAULT:
- 你好
- 嗨
- 您好
```

在 `.lg` 文件中，您还可以使用前面提到的内置函数，在发送机器人回复之前计算值或执行计算。这在处理字符串时特别有用，例如当您需要将两个或多个字符串连接或拼接成一条消息时。此外，您还可以在模板内部引用另一个模板，如下所示：

```
> 带引用的问候语
#### GreetingReply
- ${ConditionalGreeting()}, ${GreetingSuffix()}
#### ConditionalGreeting
- IF: ${time == 'morning'}
- 早上好
- ELSEIF: ${time == 'afternoon'}
- 下午好
- ELSEIF: ${time == 'evening'}
- 晚上好
- ELSE:
- 你好
- 嗨
- 您好
#### GreetingSuffix
- 您今天怎么样？
- 最近怎么样？
- 今天过得如何？
```

### 对话框“BookRestaurantTable”——确认选项

对话框中的最后一步是借助“分支：If/else”操作来检查确认结果，如图 5.28 所示。如果用户确认信息正确，则会发送一条回复，告知用户预订将由机器人执行，同时还会发送一封确认邮件。此外，预订数据会存储在一个名为`user.bookings`的数组中，该数组用于保存用户的所有预订记录，以便日后参考。如果用户未确认预订详情，机器人会询问用户是否想对预订进行修改。如果是，则对话框会简单重复，但会事先清除所有属性，以便机器人重新向用户提出所有三个问题，从而收集正确的信息。如果用户不想做任何修改，则对话框结束，这将导致调用此对话框的父对话框恢复执行。

## 将第三方 API 集成到您的聊天机器人中

机器人的一个常见用例不仅是处理问题或完成特定任务，还包括将第三方 API 集成到对话中。这对于将来自另一个系统的信息直接引入对话非常有帮助。有了这个能力，用户无需在对话上下文和另一个系统之间切换。因此，用户节省了时间，因为原本需要在另一个系统中手动获取的信息，现在可以通过机器人连接到第三方系统而纳入对话中。Bot Framework Composer 提供了在对话中任意点调用 HTTP 端点的能力，以触发其他工具中的进程或获取特定信息，这些信息随后可直接用于与用户的对话中。在您迄今为止创建的示例机器人中，一个常见的简化此解决方案使用的方法是集成天气信息，以便在用户预订餐厅座位等操作之前，告知用户天气预报。

因此，`GetWeather` 对话框用于借助第三方 Web API 收集天气信息，并获取特定城市的当前天气数据和天气预报。第一步类似于之前创建的其他对话框，因为您需要收集用户想要获取天气信息的城市信息。这是通过“设置属性”操作完成的，如图 5.45 所示，当识别到 `GetWeather` 意图时，该操作会检查 LUIS 识别器是否已从用户表述中提取了 `city` 实体。如果城市尚未被识别器提取，机器人在查询天气 API 之前应提示用户输入城市名称，并将该值存储在属性 `dialog.city` 中。

![](img/495919_1_De_5_Fig45_HTML.png)

图 5.45 GetWeather 对话框 – 设置属性并收集信息

在正确设置城市名称后，下一步是询问用户是希望将当前天气还是天气预报发送到聊天中。这也可以通过“机器人提问（选择）”操作来完成，以提供“当前天气”和“天气预报”两个选项供用户选择，如图 5.46 所示。此选择提示的结果存储在属性 `dialog.weatherChoice` 中，该属性将在后续步骤中用于确定要使用的 API 端点。

![](img/495919_1_De_5_Fig46_HTML.png)

图 5.46 GetWeather 对话框 – 输入提示

根据属性 `dialog.weatherChoice`，HTTP 请求将使用当前天气的 API 端点或天气预报的端点。如图 5.47 所示，在“发送 HTTP 请求”操作中，您可以在不同的 HTTP 方法（如 `GET`、`POST` 或 `DELETE`）之间进行选择。此外，URL 也可以由属性组成，如下所示。这样，您就可以将属性值包含在 HTTP 请求的 URL、标头或正文中。

![](img/495919_1_De_5_Fig47_HTML.png)

![](img/495919_1_De_5_Fig48_HTML.png)

此函数基本上接收输入参数，这些参数随后用于填充预定义自适应卡片中的特定值。以下代码片段基本上就是您在机器人的语言生成文件中创建此可调用函数所需的内容：

{

"列": [
{
"类型": "列",
"宽度": "auto",

{
"类型": "图像",

"大小": "小",
"altText": "大部分多云天气"
}
]
},
{
"类型": "列",

{

```

# 6. 测试聊天机器人

在第 5 章中，您学习了如何使用 Microsoft Bot Framework Composer 创建聊天机器人。在本章中，您学习了如何编写各种对话框和操作来设计对话体验。创建聊天机器人后的下一步是测试聊天机器人。这正是本章“测试聊天机器人”的主题，重点在于在本地计算机上测试和调试聊天机器人，而无需事先发布机器人。

在本章中，您将学习如何使用 Bot Framework Emulator 测试和调试聊天机器人的对话。此外，您还将了解如何在 Composer 对话框中插入跟踪事件，以向 Bot Framework Emulator 发送跟踪活动。同时，还会展示直接在模拟器中跟踪语言理解服务性能的选项。

## Bot Framework Emulator 简介

当您在本地开发聊天机器人时，无论您是使用 Composer 还是 Bot Framework SDK 和 IDE 来创建机器人，都需要在开发过程中进行测试，以查看对话体验是否符合预期。为了测试机器人的对话，可以使用桌面应用程序 Bot Framework Emulator 与机器人进行聊天。此外，您还可以使用模拟器测试远程聊天机器人，例如托管在 Azure 上的机器人。除了能够与机器人聊天外，您还可以查看用户与机器人之间交换的消息。使用模拟器的优势在于，即使您尚未将机器人发布到 Azure 订阅中，也可以在本地测试和检查聊天机器人。

注意

要下载并安装 Bot Framework Emulator，请访问 [`aka.ms/bfemulator`](http://aka.ms/bfemulator)。

作为机器人开发者，您应将模拟器的使用纳入开发流程。与其他基于 GUI 的解决方案一样，用户体验（在机器人中即对话）是用户唯一能体验到的内容。因此，在发布或分享聊天机器人之前，应彻底测试对话体验，因为如果第一印象不佳，用户将不会再使用您的聊天机器人。

## 使用模拟器测试聊天机器人

安装 Bot Framework Emulator 后，您可以通过机器人的消息端点连接到任何正在运行的机器人。消息端点通常采用以下格式的 URL：

```
http://botEndpoint/api/messages
```

在开发和本地运行机器人时，该消息端点很可能类似于：

```
http://localhost:port/api/messages
```

新创建的 Bot Framework 机器人的默认端口号为 3979，但可根据需要在机器人配置中进行更改。因此，您可以在本地计算机上通过 Visual Studio、Node.js 或其他技术运行机器人，并通过机器人 URL 或消息端点在模拟器中与其连接，如图 6.1 所示。如果您的机器人已发布到 Azure，或者机器人的应用程序设置中已包含 Microsoft App ID 和 Microsoft App 密码，则您需要在打开机器人前输入这些值。否则，由于缺少这些参数，模拟器将无法成功连接到您的机器人。

# 7. 发布聊天机器人



发布机器人很可能是聊天机器人开发生命周期中的最后一步之一，因为此操作是在您创建并本地测试聊天机器人，直到对对话体验感到满意后进行的。如果情况如此，您可以发布机器人，以便稍后将其与 Azure Bot Service 提供的通道连接，这将在第 8 章中描述。然而，本章的目标是引导您完成将 Composer 机器人发布到 Microsoft Azure 的过程。

发布过程本质上由多个步骤组成，这些步骤必须成功执行，才能使基于 Composer 的机器人在您的 Azure 环境中运行。首先，您需要在启动发布过程之前安装先决条件：

*   `Node.js`。使用版本 12.13.0 或更高版本。
*   `Azure CLI`

在安装完这两个先决条件后，您就可以开始发布在第 5 章中创建的机器人了。

## 创建 Azure 资源

在您将机器人的源代码发布到 Azure Web 应用之前，您需要在 Azure 中预配所有必需的资源，例如 Azure 应用服务计划、Azure Cosmos DB 和 Azure 机器人通道注册服务。要创建这些 Azure 资源，您需要打开一个终端实例，并导航到机器人存储在本地计算机上的位置。在该位置有一个名为 `scripts` 的文件夹，您需要在终端窗口中导航到该文件夹。在此文件夹中，您首先需要运行以下命令来安装稍后在部署过程中使用的所有依赖项：

```
npm install
```

在依赖项成功安装后，您可以启动机器人的部署过程。通常，位于机器人根文件夹中的 `README` 文件会指示您运行以下命令来预配 Azure 资源：

```
node provisionComposer.js --subscriptionId= --name= --appPassword= --environment=
```

然而，本书中描述的情况略有不同，因为我们已经在 Azure 资源组中预配了 LUIS 服务和 QnA Maker 服务。因此，您需要使用以下命令来成功预配您的 Azure 资源：

```
node provisionComposer.js --subscriptionId= --name= --appPassword= --environment= --createLuisAuthoringResource false --createLuisResource false --createQnAResource false
```

**注意：** 如果您希望将 Azure 服务预配到已有的 Azure 资源组中，您需要修改位于机器人脚本路径中的 `provisionComposer.js` 文件，以确保该脚本能够捕获现有 Azure 资源组的名称作为参数。

在此脚本成功执行后，应该会创建一个新的 Azure 资源组，其名称与您在前面的命令中指定的名称一致。该资源组应包含以下服务：

*   机器人通道注册
*   Azure Cosmos DB 帐户
*   Application Insights
*   应用服务计划
*   应用服务
*   存储帐户

这些服务对于在 Azure 中托管您的聊天机器人是必需的。`Bot Channels Registration` 服务用于首先将您的机器人连接到各个通道，并通过使用 Azure Bot Service 让用户通过这些通道与机器人通信。Azure Cosmos DB 帐户用于机器人的状态管理。Application Insights 服务用于跟踪机器人的遥测数据以及性能，这些数据可用于在 Azure 中创建仪表板。应用服务计划和应用服务用于托管机器人的运行时。这本质上是一个 Web 应用程序，其中部署了机器人的代码，并托管了机器人逻辑以及通道所连接的消息传递端点。存储帐户用作转录存储，其中保存所有聊天记录。

# 8. 将聊天机器人连接到通道

机器人开发生命周期中通常的最后一步是将聊天机器人连接到所需的通道。此过程允许您通过已启用机器人的通道启动与机器人的通信。正如前几章所述，您在开发机器人时就应该明确目标通道，因为并非所有通道的工作方式相同，也不支持相同的功能。例如，Microsoft Teams 能够处理丰富的附件（如自适应卡片），而 Twilio 通道则用于短信。由于短信无法显示富附件，因此您无法在此通道中使用自适应卡片。因此，您必须在开发周期的构建阶段就考虑目标通道，以避免后期出现较大的变更需求。

本章旨在简要描述支持的通道，并指出这些通道之间的一些差异。此外，还将详细描述将机器人连接到 Web 聊天通道和 Microsoft Teams 的过程。
