# 1. 为 JavaScript 开发者介绍 ChatGPT

## 本书面向哪些读者？

首先，本书面向那些在人工智能、自然语言处理、机器学习或深度学习方面没有任何培训或经验的 JavaScript 开发者。你可能听说过“语言模型”这个术语，但我假设这并非你日常使用的词汇。

其次，你可能熟悉（或尝试过）ChatGPT，但你不*太*理解其“底层”工作原理，也不确定如何着手将 JavaScript 和 ChatGPT 以编程方式结合使用，从而为自己的应用程序和服务“赋能 AI”。

注意

尽管 ChatGPT 家喻户晓，但其背后的公司 OpenAI 却缺乏广泛的认知度，并不那么广为人知。因此，虽然本书是关于如何在 JavaScript 应用中通过编程方式使用 ChatGPT，但我们实际使用的 API 是官方的 OpenAI REST API。因此，我们将同义使用“ChatGPT API”和“OpenAI API”这两个术语。

## 章节概述

在本章中，我们将直接深入探讨一些你可能不熟悉的术语，然后直接进入 ChatGPT Playground（游乐场）。Playground 本质上是一个为开发者构建的、基于 ChatGPT REST API 的 Web 界面。因此，一旦你熟悉了 Playground，对 ChatGPT API 的调用和响应就会变得得心应手。

## 那么，ChatGPT 到底是什么？为什么我需要使用 OpenAI API？

简单来说，ChatGPT 是由 OpenAI 开发的一组**语言模型**（深度学习算法训练的结果），能够根据给定的提示生成类似人类的文本响应。那么什么是提示？提示就是你提供给 ChatGPT 的输入文本。让我们先给 ChatGPT 一个提示，询问它对 JavaScript 设计模式的熟悉程度。前往`chatgpt.com`网站，输入清单 1-1 中的提示。

```
描述 JavaScript 开发者常用的 3 种设计模式，并说明这些模式为何重要
```

清单 1-1 提示。向 ChatGPT 提出的关于 JavaScript 设计模式的简单提示

如果你有机会在一家专业使用 JavaScript 的公司工作，那么你可能在面试中遇到过与上述提示非常相似的问题。清单 1-2 是 ChatGPT 的响应。

```
一个黑白几何标志，采用环环相扣的对称设计，类似于结或编织图案。
JavaScript 开发者通常使用设计模式来管理代码复杂性，并增强代码的可维护性、可读性和可扩展性。以下是 JavaScript 中三种广泛使用的设计模式：
1. 模块模式
模块模式用于将一组相关的方法或变量封装到单个单元或模块中，将部分内容与全局作用域隔离。此模式有助于：
•    减少命名空间污染：通过最小化全局变量的使用，此模式有助于避免命名空间污染。
•    封装：它实现了更好的封装，保护模块的状态和行为免受外部干扰。
•    代码组织：它通过将职责划分到不同的模块中，促进了组织性和结构清晰度。
2. 观察者模式
观察者模式是一种软件设计模式，其中一个称为主题的对象维护一个依赖项列表（称为观察者），并在状态发生变化时自动通知它们，通常通过调用观察者的某个方法来实现。此模式对于以下方面至关重要：
•    解耦：它将调用行为的对象与执行行为的对象解耦，从而产生更清晰、更灵活的架构。
•    事件处理：它广泛用于处理用户界面事件以及其他需要通知某些对象其他对象发生变化的场景。
•    响应式编程：促进了一种响应式编程方法，其中变化会响应事件而在系统中传播。
3. 单例模式
单例模式确保一个类只有一个实例，并提供一个全局访问点。此模式广泛用于：
•    受控访问：它提供对单个实例的受控访问，这对于管理共享资源或配置非常有用。
•    初始化保证：确保一个类只被实例化一次，这对于许多共享资源至关重要。
•    延迟初始化：通常通过延迟初始化实现，将实例的创建推迟到需要时，从而节省系统资源。
```

清单 1-2 响应。ChatGPT 对清单 1-1 中提示的响应

不错吧？ChatGPT 不仅能够列出 JavaScript 开发者常用的设计模式，还能解释它们为何重要的合理原因——全部用自然语言完成。

那么，归根结底，这对 JavaScript 开发者意味着什么？嗯，这实际上意义重大。这是因为开发者现在能够创建对自然语言文本具有一定“理解”能力的应用程序。在像 ChatGPT 这样的人工智能工具出现之前，开发者通常必须创建正则表达式（也称为“regex”）才能在文本正文中执行基本的字符和字符串模式匹配。然而，模式匹配和自然语言理解完全是两码事。
