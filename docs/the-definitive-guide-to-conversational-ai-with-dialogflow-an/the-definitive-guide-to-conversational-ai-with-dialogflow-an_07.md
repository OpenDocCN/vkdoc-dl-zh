# 关于 Dialogflow

现在，让我们进一步探讨对话式人工智能。2016 年 9 月，谷歌收购了一家名为 `API.ai` 的公司。`API.ai`（前身为 `Speaktoit` 公司）发布了一套端到端的开发套件，用于为网站、移动应用、主流消息/社交媒体平台、物联网、语音设备和联络中心构建对话界面。2017 年 10 月，该平台更名为 `Dialogflow`。`Dialogflow` 利用了人工智能的子集：自然语言理解、语音识别和命名实体识别（`NER`，用于从文本中提取值），以识别用户话语的意图、实体和上下文，从而使您的对话式用户界面能够提供高效且准确的响应。

各种规模的公司都在使用 `Dialogflow`。其用例包括：

- 内部的企业对员工聊天机器人
- 面向公众的聊天机器人，用于连接企业与客户，例如客服或销售部门
- 控制物联网设备（家庭娱乐、汽车、自助服务终端等）的聊天机器人
- 联络中心中用于处理呼入和呼出电话的机器人

`Dialogflow` 的客户包括乔治·阿玛尼、梅赛德斯-奔驰、康卡斯特、《华尔街日报》、荷兰皇家航空、易捷航空、荷兰国际集团、玛莎百货、阿霍德等。

在撰写本文时（2021 年 6 月），`Dialogflow` 拥有超过 170 万用户。`Dialogflow` 在聊天机器人社区如此受欢迎的原因在于：

- `Dialogflow` 由最先进的机器学习技术驱动。谷歌是公认的人工智能领域全球领导者，`Dialogflow` 受益于谷歌在机器学习、自然语言理解和搜索方面的资产和能力。除了内置的机器学习模型外，您还可以自行训练您的智能体，使您的对话式用户界面随着时间的推移变得更加智能。
- 使用 `Dialogflow`，您可以将对话与代码分离。由于 `Dialogflow` 提供了一个云端网页用户界面，您可以将对话和实体与应用程序/智能体代码分开。这使得您的对话式用户界面更具可扩展性；您无需开发人员即可进行更改或部署。
- 使用 `Dialogflow`，您可以通过一系列（Python）脚本比编写代码更快地构建聊天机器人。除了网页用户界面，您还可以通过启用预构建智能体（模板）和闲聊意图（为您的智能体增添更多个性）来更快地构建对话式用户界面，所有这些只需单击一次鼠标即可完成。
- 高级的履行选项和多渠道集成。`Dialogflow` 拥有超过 32 个渠道集成和软件开发工具包。因此，您可以轻松地将您的智能体与本地环境以及云环境集成，以消费来自服务的数据。借助内置的多渠道集成，您可以快速将智能体部署到各种内置渠道（如 Twitter、Facebook Messenger、Skype 或 Slack 等社交媒体渠道；Google Assistant 等语音激活助手；电话或短信服务）。或者，您可以通过使用 `gRPC`、`REST` 或适用于 `Java`、`Node.js`、`Python`、`Go`、`PHP`、`Ruby` 或 `C#` 的客户端库的软件开发工具包，将其部署到您的网站或应用程序。
- 由于 `Dialogflow` 可通过 Google Cloud 使用，因此它遵循 Google Cloud 的服务条款、服务等级协议和支持包。作为 Google Cloud 的一部分，意味着卓越的可靠性、低延迟、与超过 200 项 Google Cloud 服务（如数据分析服务和工具（例如 `BigQuery`、`Dataprep` 或 `Pub/Sub`））的轻松集成；机器学习 API（如情感检测、翻译、语音转文本转录、文本转语音合成、用于屏蔽敏感数据的数据丢失防护、用于图像光学字符识别检测的视觉 AI）或云环境（如 `Cloud Functions`、`Kubernetes`、`Compute VMs`、`Cloud Run` 或 `App Engine`）。Google Cloud 服务可以通过强大的身份访问管理、错误/调试、日志记录和监控进行控制。
- 强大的分析能力。使用数据分析来监控机器人健康状况，并更好地理解其与用户的交互。`Chatbase` 是一项 Google Cloud 服务，可帮助构建者更快地分析和优化机器人，它与 `Dialogflow` 相辅相成。结合使用它们，可以帮助构建者创建世界一流的机器人，然后使用 `Chatbase` 持续跟踪和优化它们。所有 `Dialogflow` 用户都可以直接从其控制台获得一组自动化的基本 `Chatbase` 分析数据。
- 除文本界面外的语音/语音集成。`Dialogflow` 还集成了语音识别和文本转语音功能——这对于物联网设备或与交互式语音应答电话合作伙伴的开箱即用集成非常方便。
- 多语言机器人支持。`Dialogflow` 支持超过 20 种语言。

## Dialogflow Essentials 和 Dialogflow CX

最初的 `Dialogflow` 工具最近更名了；现在它被称为 `Dialogflow Essentials`（`Dialogflow ES`），以便为新的 Google Cloud 对话式人工智能工具 `Dialogflow Customer Experience`（`Dialogflow CX`）腾出空间。

`Dialogflow CX` 将成为用于构建对话式用户界面的替代开发套件，并将与 `Dialogflow ES` 并存。谷歌将继续支持 `Dialogflow ES`，因为我们拥有庞大的用户群。为了理解谷歌为何创建另一个机器人构建器，我们首先需要从高层次了解 `Dialogflow ES` 的工作原理。

### Dialogflow Essentials 的工作原理

`Dialogflow ES` 中的意图用于对用户的意图进行分类。对于每个 `Dialogflow ES` 智能体，您可以定义许多意图，这些意图组合起来可以处理完整的对话。每个意图都可以包含参数和响应。

匹配意图也称为 `意图分类` 或 `意图匹配`。（下一章将详细介绍这些概念。）一旦匹配到意图，它可以返回响应、收集参数（实体提取）或触发 webhook 代码（履行），例如从数据库获取数据。`Dialogflow ES` 可以跟踪上下文，就像人类一样，在轮流对话中，`Dialogflow ES` 可以在第二轮和第三轮中记住上下文。这就是它能够跟踪先前用户话语的方式。这是 `Dialogflow ES` 的主要概念。

### 行业复杂性如何变化

大型企业过去几年一直在使用 `Dialogflow ES`。以下是我在谷歌观察到的一个现象。起初（2016 年），大多数对话式人工智能都是简单的聊天机器人（语音助手机器人、常见问题解答机器人等），用于网页或语音机器人，例如 Google Assistant。它通常只涉及一到两轮的轮流对话。例如，“嘿 Google，ABC 现在在播什么？”——“*《单身汉》* 晚上 8 点开始，这一集你从未看过！”

众所周知，构建对话式用户界面是一个持续的过程。当您获取到正确的聊天机器人洞察时，您会看到客户在询问什么。当您在现有智能体之上继续构建对话时，该机器人会随着时间的推移变得越来越复杂。

因此，对话**复杂性**是一个观察结果。我注意到的另一个现象是，**企业希望出现在客户所在的地方**。因此，他们的聊天机器人需要连接到更多渠道，以创造全渠道体验。公司现在不再只是构建单个聊天机器人，而是希望开发由数据湖提供支持的完整对话平台，并使用机器人流程自动化实现流程自动化。想想那些复杂的用例，例如用机器人取代您的客户服务或人力资源部门。由于呼叫中心超负荷运转以及员工因被低估的单调任务而筋疲力尽，通过聊天机器人和虚拟（语音）智能体实现自动化可以大幅削减企业成本。通过联络中心中的对话式人工智能，企业可以减少通话时间和等待时间，提供全天候服务，同时通过捕获分析数据来改进我们的流程。
