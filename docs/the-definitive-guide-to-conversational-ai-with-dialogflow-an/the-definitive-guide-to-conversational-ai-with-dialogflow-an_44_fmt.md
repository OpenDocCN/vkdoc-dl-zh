
# 在 Flutter 应用中集成 Dialogflow 的两种技术

在 Flutter 应用中集成 Dialogflow 智能体有两种技术方案。两种方案各有利弊，需要您自行判断哪种方案最适合您的业务。

## 第一种方案：直接集成 Dialogflow 包

第一种方案最为直接：使用 Dialogflow 包，将 Dialogflow SDK 直接集成到您的（移动应用）客户端中。另一种方案则是将 Dialogflow SDK 集成到后端服务器应用中，然后让您的（移动应用）客户端通过 HTTP(s) 与后端通信。

通过直接集成到 Flutter 中，您只需编写一次代码库。作为企业，您要么拥有内部的 Dart/Flutter 开发经验，要么聘请自由职业者或代理机构为您构建应用。借助 Flutter 2，您甚至可以支持桌面和 Web 应用，而不仅仅是 Android 和 iOS 应用。然而，您需要开发一种解决方案，以安全的方式提供和存储 Google Cloud 的服务账号和 API 密钥。

## 注意事项

如果处理不当，将服务账号存储在客户端资源文件夹（甚至存储在安全的密钥客户端钥匙串中）可能会很危险。首先，您需要非常小心地管理您的服务账号。您将授予您的组件哪些读取和访问权限？当您授予 Google Cloud 计算引擎管理员权限时，可以想象攻击者可能会破解并反编译您的应用。一旦他们获得对您计算引擎的访问权限，他们可能会通过例如挖矿等方式让您承担高昂的成本。即使您没有授予 Google Cloud 服务账号计算访问权限，您仍然不应暴露您的密钥。尽管对于 Dialogflow 来说，它只允许您访问项目内的聊天机器人，但如果您的最终用户在聊天中共享了个人身份信息（PII），攻击者仍有可能从聊天历史记录中读取这些数据。这可能是一个巨大的问题。

除此之外，在应用中更改服务账号密钥并不容易，因为您不希望当前用户的应用停止工作。那么，如何在移动客户端中安全地提供服务账号和密钥呢？例如，可以构建一种密钥轮换（第二组密钥）机制。比如使用 JSON Web 令牌（JWT），这是一种互联网提议的标准，用于创建带有可选签名和可选加密的数据，其有效载荷包含断言某些声明的 JSON。令牌使用私有密钥或公钥/私钥对进行签名。或者，您也可以在服务器上运行部分集成。

## 第二种方案：集成到后端/服务器应用

在后端/服务器应用中集成 Dialogflow SDK 需要您用另一种编程语言编写应用代码。对于企业而言，这通常意味着您需要与另一个开发团队或代理机构合作，并且必须协同工作。这种方案还意味着您需要支付服务器计算成本，因为您需要将代码库托管在某个地方。其优势在于，您不会将服务账号暴露给客户端，从而使该方案非常安全。

采用这种方法的另一个显著优势是，当您想要构建一个需要支持多种集成渠道的聊天/语音平台时。对于创建多渠道或全渠道体验，这是正确的做法，因为您只需在一个地方维护 Dialogflow 集成，将每个集成视为其自己的渠道。如果您愿意，甚至可以构建成能够在 Dialogflow Essentials 和 Dialogflow CX 智能体之间切换的方式。

## 将 Dialogflow SDK 直接集成到您的 Flutter 应用中

当您在 `pub.dev` 上搜索 Dialogflow 时，您会发现许多开源的 Dialogflow Flutter 包，包括我自己的：`dialogflow_grpc`。Dialogflow gRPC 包利用了 Dialogflow 的 gRPC 库。只有使用 gRPC 库才能实现流式传输，如果您想利用 Dialogflow 内置的语音转文本和文本转语音功能的话。值得一提的是，底层的 gRPC API 也比通常用于将 Dialogflow 与 Dart 和 Flutter 集成的 REST API 更快。

出于演示目的，我们将 Google Cloud 服务账号包含在资源文件夹中。正如我们现在所知，对于生产环境的应用，您必须制定更安全的策略。

使用 `dialogflow_grpc` 依赖项编辑 `pubspec.yaml` 文件。并指向您的服务账号（例如 `assets/credentials.json`），如清单 11-12 所示。

```yaml
dependencies:
  dialogflow_grpc: any
```
