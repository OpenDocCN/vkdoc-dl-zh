# 5. 深入实现细节

现在你已经熟悉了 Oracle 数字助理的背景、功能以及如何设计数字助理，是时候了解一些基本概念了。在本章中，你将首先了解 Oracle 数字助理（ODA）环境。然后，你将基于第 3 章描述的用例，开始与数字助理 `Travvy` 的旅程。每一节中，你将先了解概念，然后针对该用例实现这些概念。

## 处理意图与表述

在本节中，您将学习如何：

- 创建意图
- 向意图添加表述
- 导出意图
- 导入意图
- 训练和测试意图

在本书的第 2 章中，您已经了解了意图和表述的相关概念。
快速回顾一下，`意图` 可以描述为您打算通过技能执行的一项工作或任务。为了描述这些意图，您需要将它们与示例句子关联起来。这些描述意图的示例句子被称为 `表述`。
例如，如果您创建了一个 `PizzaSkill`，您可能希望使用该技能来订购披萨。为此，您可以添加一个意图，比如 `OrderPizza`，它允许用户通过该技能订购披萨。为了让技能识别用户的意图，您只需向技能中添加一些示例表述，例如“帮我订一份披萨。”当用户与技能交互并说出/写下“帮我订一份披萨”时，技能将能够根据该表述解析出具体的意图，在本例中就是 `OrderPizza`。您添加的表述越多，您的技能就越健壮。
您的表述应该是完整且有意义的句子，但长度不要超过 255 个字符。较短的表述通常能带来更好的结果，因为它们往往更简洁，填充词更少。此外，表述不应由单个单词构成。

### 提示

最初，您至少应该为定义的每个意图创建十到十五条表述。这能确保更好的意图解析效果。当您准备将技能部署到生产环境时，您将需要更多的表述。为意图识别表述是一个迭代过程——如果做得好，会带来良好的结果。
每个技能必须至少关联两个意图。因此，如果以 `PizzaSkill` 为例，它应该包含两个意图，比如 `OrderPizza` 和 `CancelPizza`。

### 训练与测试意图

创建或更新意图后，你需要训练技能，以便模型根据你的更改进行训练。此训练基于不同的机器学习算法，这些算法会使用相关的意图和表述来训练你的技能。当用户与你的技能交互时，此训练可确保你的技能识别出特定意图并做出相应响应。

## 使用实体

在处理意图和表述时，你了解到意图是你打算执行的特定操作，而表述是描述这些意图的有意义的句子。实体是可变元素，为意图提供上下文。简单来说，实体是你在对话过程中向用户寻求的那些值，你将根据这些值来定义机器人的结果或流程。

例如，对于你的 `FindTrip` 技能，如果用户说“我想预订一次旅行”，那么你知道用户的意图是预订旅行，但与此同时，这是一个非常笼统的陈述。相反，如果用户说“我想预订一次去美国的徒步旅行”，那么上下文就清晰得多。所以，用户想预订旅行，但去哪个地点？答案是去美国。这个有助于为意图提供恰当含义的地点，就被称为实体。因此，如果用户提供了地点，你就可以为他预订特定的旅行，否则你需要询问他地点，以便为他提供合适的预订选项。

实体分为两类：

1. 内置实体
2. 自定义实体

内置实体或系统实体是 Oracle 作为数字助手实例的一部分提供给你的实体。自定义实体是你根据自己的需求创建的实体。

## 内置实体

内置实体分为两类：

1. 简单实体
2. 复杂实体

### 简单实体

简单实体基于原始类型，如字符串和整数。简单实体的示例包括：

* `NUMBER`
* `EMAIL`
* `YES_NO`

### 复杂实体

另一方面，复杂实体基于特定属性。表 5-2 展示了复杂实体及其属性。

表 5-2 复杂实体

| 实体 | 属性 |
| --- | --- |
| `ADDRESS` | `city`、`houseNumber`、`road`、`originalString` |
| `DATE` | `originalString`、`date` |
| `TIME` | `hrs`、`mins`、`secs`、`"hourFormat":"PM"` |
| `DURATION` | `startDate`、`endDate`、`originalString` |
| `SET` | `minute`：范围为 {0–59}，`hour`：范围为 {0–23}，`dayOfTheMonth`：范围为 {1–31}，`monthOfTheYear`：范围为 {1–12}，`dayOfTheWeek`：范围为 {0–6}，以 0 表示星期日，`year` |
| `CURRENCY` | `amount`、`currency`、`totalCurrency` |
| `PHONE_NUMBER` | `phoneNumber`、`completeNumber` |
| `URL` | `protocol`、`domain`、`fullPath` |

## 自定义实体

自定义实体是特定于技能的实体，你将创建它们来存储或检索自定义值。例如，对于金融机器人，你可以考虑使用 `AccountType` 实体，它可以具有 `Savings` 和 `Current` 账户等值。或者对于 `PizzaBot`，`CrustType` 可以是 `Thin`、`Thick` 或 `Medium`。

以下是自定义实体的类型：

* `派生实体`：顾名思义，它们派生自内置实体或你定义的自定义实体。
* `值列表实体`：这些实体基于预定义的值列表。
* `实体列表实体`：这些实体是实体的超集，并基于其他实体的列表。
* `正则表达式实体`：这些实体基于正则表达式（regex）来严格匹配特定模式。
* `复合包实体`：这些实体是一组相关实体，可以解析到特定领域或特定功能。你可以通过不同方式配置复合包实体。你可以提示用户输入构成复合包的每个实体的值，也可以提示用户输入特定实体，并根据提供的实体值推导出复合包中的其余实体。

## 使用对话流

到目前为止，你已经熟悉了意图、表述和实体。现在是时候在对话流中使用它们了，这样你的技能就能具备处理对话的能力。对话流是设计技能对话的地方。技能应该如何问候、应该向用户询问什么、以及如何根据用户输入做出反应——所有这些都通过对话流中的各种组件来处理。因此，可以说对话流定义充当了任何技能的模型。

对话流是使用 Oracle Bots 标记语言 (OBotML) 设计的。OBotML 基于 YAML，这是一种表达数据和数据关系的人性化方式。

你可以通过点击左侧导航栏中的“流” (![流图标](img/479330_1_En_5_Figy_HTML.jpg)) 来访问技能的对话流。在继续之前，理解构成对话流的各个部分至关重要。请参见图 5-26。

![对话流部分](img/479330_1_En_5_Fig26_HTML.jpg)
图 5-26 对话流部分

对话流大致可以分为三个部分：

1. 头部：包含 OBotML 平台版本和对话流名称。
2. 上下文：包含上下文和变量，也称为对话流变量，这些变量在流程中使用。
3. 状态：包含各个状态、动作及其结果。此部分包含技能交互和响应的主要逻辑。

由于 OBotML 遵循 YAML 标准，因此在对话流中保持正确的缩进非常重要。如果你编写的行没有正确缩进，在设计时自动保存期间将会收到错误。

> **注意**：每个缩进的长度等于两个空格。

流程的头部部分相当直接。它包含你的机器人的元数据信息。但涉及到上下文和状态时，你需要更仔细地查看。

### 上下文部分

如前所述，你在上下文部分为技能定义了全局作用域的变量，然后可以在技能对话流的各个状态中使用这些变量。上下文部分中的变量定义如下：

`variableName`: “`variableType`”;

显然，`variableName` 是你给变量起的名称。`variableType` 是变量的类型，可以是：

* 原始类型（`string`、`int`、`boolean`、`double` 和 `float`）
* 内置实体（`CURRENCY`、`ADDRESS` 等）
* 自定义实体（如前面章节创建的 `TripLocation`）
* `nlpresult`，来自自然语言处理引擎的结果
* `resourcebundle`，用于你定义的引用技能中资源包的变量

这些变量在对话结束时或被显式重置时总是会被清除。

### 状态部分

状态是定义技能整个流程的地方。在这里，你定义技能应该执行的所有活动和动作。你的流程通常由许多状态组成。这些状态调用组件，组件为状态执行任务或工作。

> **注意**：一个状态只能调用一个组件。

流程总是从上到下执行。这意味着在状态部分，你定义的第一个状态首先被执行，第二个状态接下来被执行，依此类推。这是默认行为，除非你显式设置转换到另一个状态。

此外，强烈建议在你的流程中设置一个默认转换，以优雅地处理因不可预见的用户输入而可能产生的任何意外异常/错误。

在开始使用对话流之前，你还需要了解另外两件事：Apache FreeMarker 模板语言 (FTL) 和对话流中的组件。

首先，我们来处理 Apache FreeMarker 模板语言及其工作原理。

### Apache FreeMarker 模板语言

为简单起见，在 ODA 环境中，您使用 Apache FreeMarker 模板语言 (FTL) 来设置和访问变量的值（以及它们的属性，如果有的话）。如果您想了解更多关于 FTL 的信息，可以访问其官方网站 [`freemarker.apache.org/`](https://freemarker.apache.org/)。

在对话流中使用 FTL 的语法相当直接。您只需将想要使用 FTL 处理的所有内容包裹在 `${…}` 中。您可以在组件的属性定义中使用 FTL（组件将在本章下一节中描述）。例如，如果您定义了一个名为 `myVariable` 的变量，并想读取它的值，那么您只需使用：

```ftl
${myVariable.value}
```

或者，如果您定义了一个变量，它是一个类型为 `ADDRESS` 的复杂实体，比如 `userAddress`，那么您可以使用 FTL 来提取城市，如下所示：

```ftl
${userAddress.value.city}
```

FTL 的用途不仅限于此。您还可以使用 FTL 在流程中编写复杂的表达式。一个非常常见的场景是在 `System.CommonResponse` 组件中使用 FTL，您可以在其中使用 `if`、`else` 或 `elseif` 指令定义条件。其语法如下：

```ftl
<#if yourCondition>result1<#else>result2</#if>
<#if firstCondition>result1<#elseif secondCondition>result2<#elseif thirdCondition>result3</#if>
```

当然，FTL 的使用并不仅限于 `CommonResponse` 组件，您可以在整个对话流中使用它们，用于在机器人响应中赋值或进行评估。

此外，您还可以使用 FTL 支持的各种操作，例如针对字符串、数字、日期时间值等的操作。要了解更多信息，请参考链接 [`freemarker.apache.org/docs/ref.html`](https://freemarker.apache.org/docs/ref.html)。

对 FTL 进行冗长的讨论，涵盖所有这些用例和操作，可能会超出本书的范围。因此，我们在此结束本节。任何 FTL 的进一步用法将在需要时进行解释。

### 对话流中的组件

您想在技能中实现的任何功能都来自组件。无论您是想显示卡片或列表、根据用户输入执行特定操作、显示任何输出、状态转换、处理错误等，甚至是想执行某些特定任务（如调用 REST 调用来处理用户响应），您都将借助组件来完成所有这些操作。

组件可以大致分为两类：

- 内置组件
- 自定义组件

顾名思义，内置组件是 Oracle 作为数字助手框架的一部分提供给您的组件，供您在技能中使用。这些组件足以执行大多数技能操作。您可以在流程中轻松识别这些组件，因为它们总是在 `states` 部分的 `component` 下以 `System.` 开头。请参考图 5-26。

如果在任何时候，您想在技能中执行某些无法使用内置组件完成的操作，那么您可以定义自己的组件。您为执行特定任务而创建的这些组件称为自定义组件。

### 内置组件

如前所述，这些是 ODA 框架中预定义的组件。要将内置组件添加到您的流程中，您需要点击编辑器顶部的 `+ Components` 按钮。这将打开一个新窗口，如图 5-27 所示，为您提供多种组件类型可供选择。
