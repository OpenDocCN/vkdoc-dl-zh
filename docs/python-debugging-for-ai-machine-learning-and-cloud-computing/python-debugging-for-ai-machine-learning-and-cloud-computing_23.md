从上面的输出中，你看到线程 #3 正在等待由线程 #2（`0x7f7e56743700`）拥有的 `RLock 0x7f7e56891d80`。你还看到线程 #2 正在等待由线程 #3（`0x7f7e55f42700`）拥有的 `RLock 0x7f7e56891f30`。基于这种相互等待，你推测存在**死锁**。

## 调试架构

由于源代码很小（清单 12-1），让我们进行**事后**调试，采用**现场**和**纸上**方式。

```
### deadlock.py
import time
import threading
cs1 = threading.RLock()
cs2 = threading.RLock()
def thread_func1():
with cs1:
time.sleep(1)
with cs2:
pass
def thread_func2():
with cs2:
time.sleep(1)
with cs1:
pass
def main():
thread1 = threading.Thread(target=thread_func1)
thread2 = threading.Thread(target=thread_func2)
thread1.start()
thread2.start()
thread1.join()
thread2.join()
if __name__ == "__main__":
main()
清单 12-1
一个说明两个线程间死锁的简单脚本
```

你立即看到了线程中相反的锁获取顺序。如果你修正它（清单 12-2），脚本将运行完成而不会冻结：

```
def thread_func1():
with cs1:
time.sleep(1)
with cs2:
pass
def thread_func2():
with cs1:
time.sleep(1)
with cs2:
pass

```

## 异常与死锁

`with` 是 `acquire` 和 `release` 方法的语法糖。如果直接使用，或者在使用具有相同 acquire 和 release 语义的第三方及原生操作系统同步对象时，在存在**已处理异常**的情况下，你可能会遇到同步问题（清单 12-3）。异常会跳过锁的 `release` 方法，从而导致**死锁**。在这种情况下，我建议在异常处理代码中添加日志记录，以便对异常处理过程具有可见性。

```
### deadlock-exception.py
import time
import threading
import logging
cs1 = threading.RLock()
cs2 = threading.RLock()
def foo (data):
if data: raise "Exception"
def thread_func1():
try:
cs1.acquire()
foo(1)
cs1.release()
except:
logging.error("Exception logged")
time.sleep(2)
cs2.acquire()
cs2.release()
def thread_func2():
cs2.acquire()
cs1.acquire()
time.sleep(3)
cs1.release()
cs2.release()
def main():
thread1 = threading.Thread(target=thread_func1)
thread2 = threading.Thread(target=thread_func2)
thread1.start()
thread2.start()
thread1.join()
thread2.join()
if __name__ == "__main__":
main()
清单 12-3
一个说明在存在已处理异常的情况下两个线程间死锁的脚本
```
