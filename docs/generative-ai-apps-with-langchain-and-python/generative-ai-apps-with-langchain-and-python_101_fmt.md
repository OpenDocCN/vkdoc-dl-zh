## RAG 的重要性

让我们谈谈为什么 RAG（检索增强生成）如此重要。您看，许多 LLM 应用程序需要模型训练集中未包含的用户特定数据。这带来了几个挑战：

1. **最新信息**：当您想询问模型知识截止日期之后发生的事件时，模型无法给出有意义的响应。它甚至可能导致幻觉。

2. **私有文档**：当您需要基于自己的私有文档的答案时，传统的 LLM 无法胜任。

3. **企业数据安全**：对于您的企业，您需要保护敏感信息的安全。

这就是 RAG 的用武之地，因为它允许您在内容生成步骤期间检索外部数据并将其传递给 LLM。它通过以下几种方式应对这些挑战：

1. **最新信息**：RAG 可以通过拉取最新数据来确保您的响应是最新的。

2. **个性化知识**：它可以访问您的私有文档以提供量身定制的答案。

3. **数据隐私与安全**：至关重要的是，RAG 有助于将高度珍视的企业数据大部分保留在 OpenAI 等 LLM 供应商的掌控之外。具体方式如下：

   - **选择性数据共享**：仅将回答特定问题所需的最少量数据发送给 LLM。其余数据安全地保留在您公司的控制之下。

   - **数据驻留**：您公司的大部分数据保留在您自己的服务器或选定的云基础设施上，永远不会离开您公司的安全边界。

   - **上下文检索**：您可以设计您的 RAG 系统，使其仅检索和发送文档的非敏感部分，从而保护您的敏感信息。

   - **合规性**：通过限制敏感数据暴露给第三方服务，您可以确保遵守数据保护法规。

因此，您可以在利用先进的 LLM 能力和维护严格的数据安全协议之间取得平衡。这就是为什么 RAG 对于寻求负责任地采用 AI 解决方案的企业来说是一项关键技术。

## RAG 如何工作？

让我们看一下图 7-1 中 RAG 过程的视觉表示。

**图 7-1.** RAG 过程

正如您从图 7-1 中看到的，一切都始于源，即您收集宝贵数据的地方。这个源可以是大量的文档集合、网页或您想要使用的任何其他文本数据。对于每个源来说，这是一个一次性的过程。

需要注意的是，数据摄取和处理不是一个一次性的过程，而是一个需要谨慎管理的持续过程：

- **初始设置**：数据源的初始摄取是设置 RAG 系统的第一步。

- **定期更新**：随着您的源数据发生变化或增长，请注意您需要相应地更新已处理的数据。这可能涉及：

  - 对新内容或修改内容进行增量更新

  - 在发生重大更改或为了保持一致性时进行完全重新处理

**加载阶段**：此阶段涉及从已识别的源提取数据并将其带入您的处理环境。关键步骤包括：

- **提取**：从原始源检索原始数据

- **初始摄取**：将数据存储在您系统的工作环境中

- **验证**：对完整性和完整性进行基本检查

- **日志记录**：记录有关已加载数据的元数据

目标是使原始数据可用于进一步处理。

**转换阶段**：然后我们进入转换阶段，在此阶段您预处理并准备数据以供后续步骤使用。您将参与诸如清理文本、删除无关信息甚至将数据拆分成更小的块等任务。

**嵌入阶段**：接下来是 RAG 中真正的游戏规则改变者，即嵌入阶段。这是您使用文本嵌入模型（如 OpenAI 的 `text-embedding-ada-002`）将转换后的数据转换为称为嵌入的数值表示的地方。嵌入就像一种捕获文本语义含义的秘密代码。通过嵌入您的数据，您使模型能够理解不同信息片段之间的关系和相似性。

**存储阶段**：在嵌入阶段之后，您将嵌入存储在一个称为向量数据库或索引的特殊数据结构中。您将使用像 Pinecone、Chroma、Milvus 或 Qdrant 这样的向量数据库。这允许您在需要时快速检索相关信息。

**检索阶段**：最后，在检索阶段，您将向 RAG 模型提供一个查询或提示词以生成文本。使用 LangChain，您将使用相同的嵌入模型为该问题生成一个嵌入。然后，您使用问题的嵌入从您的向量数据库中找到最相似的块嵌入。这是通过根据向量与问题嵌入的余弦相似度或欧几里得距离对向量进行排名来完成的。排名最高的向量代表与问题最相关的块。然后，模型使用该查询从存储的嵌入中检索最相关的信息。

一旦检索到相关信息，模型就会使用它来生成新的文本，该文本融合了检索到的知识以及其预先存在的知识。

通过遵循此流程，您可以构建一个强大的问答系统，该系统充分利用了两者的优势——语言模型固有的知识和从外部源检索相关信息的能力。

### RAG 用例示例

让我分享一个例子使其更具体。想象一下，您正在构建一个聊天机器人，用于回答有关您公司产品的问题，例如详细规格、用户手册或销售数据。在这种情况下，您希望答案仅限于您的产品。这就是 RAG 的闪光点，因为它从您的自定义数据中提供准确的信息，并避免任何通用内容。

以下是您将如何完成 RAG 过程：

1. 将您的自定义数据（例如用户手册、产品规格）加载到您的系统中。

2. 通过清理和预处理数据来转换数据，以确保一致性和质量。

3. 嵌入转换后的数据以捕获其语义含义。嵌入允许精确且上下文感知的检索。

4. 将嵌入存储在向量数据库中，以便高效快速地检索。

现在，当用户提出诸如“产品 X 的安装步骤是什么？”之类的问题时，RAG 模型就会开始行动。RAG 模型不会提供通用 LLM 可能生成的通用答案，而是会：

- 从您的自定义数据中检索最相关的信息

- 基于该数据生成准确且特定于上下文的响应

这确保了聊天机器人提供精确且相关的答案，这些答案严格与您的产品相关。您将获得一个信息丰富且上下文感知的答案，它结合了两者的优势：存储在外部数据源中的海量知识和现代语言模型的语言理解能力。

## 文档加载器

任何检索系统的核心都是文档加载器。文档加载器允许您从各种源加载文档，例如 HTML、PDF 甚至代码文件。您可以从 LangChain 提供的 100 多种不同的文档加载器中进行选择，以集成来自简单 `.txt` 文件、公共网站的 HTML、YouTube 视频的转录、来自私有 S3 存储桶的 PDF 文件或来自各种存储库的代码片段等源。您甚至可以与 AirByte 和 Unstructured 等流行提供商集成。

这些文档可以存储在私有 S3 存储桶中，甚至存储在公共网站上。

### 文档加载器实战

使用文档加载器很容易。您将首先从各种源加载数据，并将其转换为文档对象。文档就像一个整洁的小容器，包含一段文本及其相关的元数据。

每个加载器都有两个关键方法：`Load` 和 `Load and split`。`Load` 方法从配置的源加载文档。`Load and split` 方法更进一步，它加载文档，然后使用您选择的文本分割器对其进行分割。

一些文档加载器还提供 `Lazy load` 方法。此方法允许您惰性地将文档加载到内存中，您会发现这在处理大型数据集时非常有用。它通过按需获取文档来帮助您。
